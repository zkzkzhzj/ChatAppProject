# ADR-005: 게스트 마을 상태 관리 — 서버 즉석 생성 (비영속)

## 상태

확정 (2026-04-07)

---

## 맥락

게스트 유저는 회원가입 없이 마을에 입장할 수 있다.  
이때 캐릭터와 공간을 어떻게 관리할 것인가?

---

## 선택지

**A. DB 저장 (회원과 동일)**
- 게스트 토큰 발급 시 임시 character/space를 DB에 생성
- 회원가입 시 데이터 인계
- 단점: DB에 게스트 쓰레기 데이터 누적, 만료 처리 복잡

**B. Redis TTL 저장**
- 게스트 세션 동안 Redis에 캐릭터 상태 보관
- 단점: Redis 의존, 구현 복잡도, 새로고침해도 유지되어 사용자 기대와 다를 수 있음

**C. 서버 즉석 생성 (비영속)**
- `GET /api/v1/village/characters/me` 호출 시 기본 캐릭터를 그 자리에서 생성해 반환
- DB, Redis 저장 없음
- 새로고침 → 동일한 기본값으로 재생성

---

## 결정

**C. 서버 즉석 생성** 채택.

이유:
- 게스트 경험 의도: 마을을 "구경"하는 수준. 커스터마이징이나 데이터 지속성이 없어도 된다.
- 새로고침 시 초기화는 의도한 동작이다. 게스트에게 지속성을 기대하지 않는다.
- DB/Redis에 게스트 상태를 저장하면 만료 처리, 데이터 정리 등 운영 복잡도가 생긴다.
- WebSocket 실시간 이동 시 위치 추적은 별도 레이어(Redis, in-memory)에서 처리한다. 이 ADR의 범위가 아님.

---

## 동작 정리

| 엔드포인트 | MEMBER | GUEST |
|-----------|--------|-------|
| `GET /api/v1/village/characters/me` | DB 조회 (비동기 생성 전 404 가능) | 기본 캐릭터 즉석 생성 |
| `GET /api/v1/village/spaces/me` | DB 조회 | 403 (개인 공간 없음) |

---

## 향후 고려

- 게스트가 마을 WebSocket에 참여할 때 위치 관리는 Redis TTL로 처리 예정 (Phase 3+)
- 회원 전환 시 게스트 캐릭터 외형을 이어받는 기능이 필요하면 별도 ADR로 결정
