# ADR-006 — Cassandra를 메시지 저장소로 선택

## 상태

Accepted (Phase 3, 2026-04-08)

---

## 배경

채팅 메시지는 다른 데이터와 성격이 근본적으로 다르다.

- **쓰기 패턴**: 메시지는 항상 append-only다. 기존 메시지를 수정하거나 삭제하는 일이 거의 없다.
- **조회 패턴**: 특정 채팅방의 최근 N개 메시지를 시간 역순으로 읽는 것이 유일한 쿼리다.
- **규모**: 서비스가 성장하면 메시지 수는 다른 어떤 테이블보다 빠르게 증가한다.

PostgreSQL에 메시지를 저장하면:
- 시간이 지날수록 단일 테이블에 수억 건이 쌓인다.
- `WHERE chat_room_id = ? ORDER BY created_at DESC LIMIT N` 쿼리는 인덱스를 써도 결국 랜덤 I/O가 발생한다.
- 파티셔닝으로 해결할 수 있지만 운영 복잡도가 크게 올라간다.

---

## 결정

메시지 저장소로 Cassandra를 선택한다.

### 파티션 설계

```
Partition Key : chat_room_id     → 같은 채팅방 메시지가 같은 노드에 모임
Clustering Key: created_at DESC  → 최신 메시지가 먼저 오도록 정렬
               id DESC            → 동일 시각 메시지 간 정렬 보장
```

이 설계로 `WHERE chat_room_id = ? ORDER BY created_at DESC LIMIT N` 쿼리가 단일 파티션 순차 읽기로 처리된다. Cassandra의 가장 빠른 쿼리 패턴이다.

---

## 트레이드오프

| 항목 | 내용 |
|------|------|
| 운영 복잡도 | PostgreSQL에 더해 Cassandra 클러스터를 별도로 운영해야 한다. |
| 트랜잭션 없음 | 메시지 저장과 다른 데이터 변경을 한 트랜잭션으로 묶을 수 없다. 현재 구조에서는 문제 없음 (메시지는 독립 저장). |
| 학습 비용 | Spring Data Cassandra 설정, PrimaryKeyClass 구조 등 초기 학습 비용이 있다. |
| 조회 제약 | 채팅방 ID 기준 외의 조회(특정 유저의 전체 메시지 등)는 별도 인덱스 또는 다른 저장소 필요. |

---

## 거부된 대안

**PostgreSQL 단일 저장소**: 단기적으로 더 단순하지만, 서비스 성장 시 메시지 테이블이 병목이 된다. 나중에 마이그레이션하는 비용이 더 크다고 판단.

**Redis**: 인메모리라 영속성이 약하다. 메시지는 영구 저장이 필요하므로 부적합.
