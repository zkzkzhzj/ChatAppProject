# TDD Cycle

## 목적

새 동작을 테스트로 먼저 고정하고, 최소 구현 후 리팩터링한다.

## 사용 조건

- 새 기능을 추가한다.
- 버그 재현 조건이 명확하다.
- 도메인 규칙, 권한, 동시성, 멱등성처럼 회귀 비용이 크다.
- 테스트가 구현보다 더 명확한 요구사항 역할을 할 수 있다.

## 절차

1. 기대 동작과 실패 조건을 테스트 이름으로 표현한다.
2. 실패하는 테스트를 먼저 작성한다.
3. 테스트가 실제로 실패하는지 확인한다.
4. 통과에 필요한 최소 구현만 작성한다.
5. 관련 테스트를 통과시킨다.
6. 중복과 이름을 정리하되 동작은 바꾸지 않는다.
7. 전체 영향 범위에 맞는 테스트를 다시 실행한다.

## 예외

- 문서만 변경하는 작업
- 빌드 설정이나 의존성 패치처럼 별도 검증이 더 직접적인 작업
- 테스트 인프라 자체가 없어 먼저 기반 작업이 필요한 경우

예외를 적용하면 완료 보고에 이유와 대체 검증을 적는다.
