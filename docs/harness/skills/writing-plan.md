# Writing Plan

## 목적

승인된 설계를 작고 검증 가능한 구현 단계로 나눈다.

## 사용 조건

- 새 기능, 새 API, DB 변경, 하네스 변경처럼 변경 범위가 넓다.
- 여러 계층이나 여러 파일을 건드린다.
- 병렬 에이전트나 Critic Gate가 필요한 작업이다.
- 1회 수정으로 끝내기 어렵다.

## 절차

1. 설계 요약과 성공 조건을 확인한다.
2. 변경 파일 후보를 계층별로 나눈다.
3. 각 단계를 커밋 가능한 작은 작업으로 쪼갠다.
4. 각 단계에 검증 방법을 붙인다.
5. 필요한 Agent와 Critic을 단계별로 표시한다.
6. 티켓이 너무 크면 시작 전에 티켓 분할 여부를 결정한다.
7. 장기 작업이면 `track-start.md`로 넘긴다.

## 전이 규칙

계획은 사용자가 별도로 요청해야 시작되는 단계가 아니다.

- 승인된 설계가 있고 구현 범위가 둘 이상의 파일/계층을 건드리면 자동으로 작성한다.
- 계획 작성 중 독립 검토가 필요하면 `parallel-agent-dispatch.md`를 호출한다.
- 계획이 완료되면 트랙이 필요한지 판단하고, 필요하면 `track-start.md`로 전이한다.
- 트랙이 필요 없고 작업 트리가 안전하면 첫 작업 커밋 구현으로 넘어간다.
- 이슈 번호, 브랜치명, 배포 범위처럼 외부 결정이 필요한 경우에만 사용자에게 묻는다.
