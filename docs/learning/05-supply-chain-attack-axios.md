# npm 공급망 공격 대응: axios 1.14.1 사례

> 작성 시점: 2026-04-04
> 맥락: 프론트엔드 의존성 검토 중 axios 1.14.1이 악성 버전으로 확인됨

---

## 무슨 일이었나

2025년경, `axios@1.14.1`이 npm에 게시되었는데 이 버전에 악성 코드가 포함되어 있었다.
npm 공급망 공격(supply chain attack)의 대표적인 사례다.

공격 방식:

1. 공격자가 정식 패키지의 버전 번호를 패치 버전(1.14.0 → 1.14.1)처럼 올려서 npm에 게시
2. `"axios": "^1.14.0"` 같은 범위 지정(`^`)을 쓰는 프로젝트는 자동으로 1.14.1을 받게 됨
3. 악성 코드 실행

---

## `^` 버전 범위의 위험성

npm(과 대부분의 패키지 매니저)에서 `^`는 마이너/패치 버전 업그레이드를 허용한다:

```json
"axios": "^1.14.0"
// 실제로 설치되는 버전: >=1.14.0 <2.0.0
// 즉, 1.14.1, 1.15.0, 1.99.9 등이 자동으로 설치될 수 있다
```

`npm install` 또는 CI에서 의존성을 새로 설치할 때마다 최신 패치 버전을 받아온다.
이 편의성이 공급망 공격의 진입점이 된다.

---

## 대응 방법

### 1. 버전 고정 (채택)

```json
"axios": "1.14.0"  // ^ 없음 = 정확히 이 버전만
```

`^`나 `~`를 제거하면 그 버전만 설치된다.

### 2. lockfile 커밋

`package-lock.json`(npm) 또는 `yarn.lock`을 반드시 git에 커밋한다.
lockfile이 있으면 `npm ci`는 lockfile에 명시된 정확한 버전을 설치한다.
lockfile 없이 `npm install`만 믿으면 범위 내 최신 버전이 설치될 수 있다.

### 3. `npm audit`

```bash
npm audit
```

알려진 취약점이 있는 패키지를 보고해준다. CI 파이프라인에 포함시키면 자동 감지 가능.

---

## 교훈

- 유명 패키지라도 특정 버전이 악성일 수 있다
- `^` 범위 지정은 편리하지만 자동 업그레이드를 허용한다
- 프로덕션 의존성은 정기적으로 감사하고, lockfile을 항상 커밋해야 한다
- CI에서 `npm ci`를 쓰면 lockfile 기반으로 설치되어 예상치 못한 업그레이드를 방지한다

---

## 이 프로젝트의 조치

```json
// 변경 전
"axios": "^1.14.0"

// 변경 후
"axios": "1.14.0"
```

1.14.0은 공격 이전 안전한 버전으로 확인됨.
이후 axios 업그레이드 시에는 릴리스 노트와 npm advisory를 확인 후 명시적으로 버전을 올린다.
