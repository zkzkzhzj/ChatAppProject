# 32. 웹 기반 2D/3D 게임 엔진 비교 — Phaser.js가 2D 인터랙티브 마을에 가장 적합한 이유

> 작성 시점: 2026-04-15
> 맥락: "마음의 고향" 프로젝트에서 2D 인터랙티브 마을(캐릭터 이동, 공간 꾸미기, NPC 대화)을 구현하기 위한 게임 엔진 선정

---

## 배경

이 프로젝트는 2D 탑다운 뷰의 마을에서 유저 캐릭터가 돌아다니고, 공간을 꾸미고, 이웃(다른 유저 또는 AI NPC)과 대화하는 서비스다. Gather Town과 유사한 형태의 인터랙티브 2D 공간이 필요하다.

웹 기반으로 구현해야 하므로 JavaScript/TypeScript 게임 엔진 중에서 선택해야 했다. 핵심 요구사항은 다음과 같다:

- **32x32 타일맵 기반 2D 렌더링** (Tiled 에디터 호환)
- **캐릭터 스프라이트 + 4방향 이동 애니메이션**
- **물리엔진** (충돌 감지 — 벽, 가구 등)
- **Next.js(React)와의 통합** (채팅 UI, 인벤토리 등은 React로 처리)
- **낮은 학습곡선** (1인 개발, 백엔드가 주력)

## 선택지 비교

### 3D 엔진 (왜 제외했는가)

| | Three.js | Babylon.js | React Three Fiber |
|--|---------|-----------|-------------------|
| 핵심 개념 | 3D 렌더링 라이브러리. 카메라, 조명, 머티리얼 등 3D 개념 필수 | MS 지원 올인원 3D 엔진. 물리, GUI, 파티클 내장 | Three.js의 React 래퍼. JSX로 3D 씬 구성 |
| 2D 가능 여부 | 가능하지만 오버킬. OrthographicCamera + PlaneGeometry로 2D 구현 | 동일. 2D 스프라이트 모드 있지만 본질은 3D | Three.js와 동일 |
| 타일맵 | 직접 구현 필요 | 직접 구현 필요 | 직접 구현 필요 |
| 왜 제외? | 이 프로젝트는 순수 2D. 3D 개념을 배우고 관리하는 비용 대비 얻는 게 없음 | 동일 | 동일 |

3D 엔진은 2D 타일맵 게임에서는 **복잡도만 높이고 이점이 없다**. 나중에 2.5D(아이소메트릭) 또는 3D로 전환하고 싶다면 그때 검토해도 늦지 않다. 초기 단계에서 "나중을 위해" 3D 엔진을 쓰는 건 전형적인 오버엔지니어링이다.

### 타일맵/레벨 디자인 도구

| | Tiled | LDtk |
|--|-------|------|
| 역사 | 10년 이상. 업계 표준 | 2020년~. Dead Cells 디렉터가 개발 |
| UI/UX | 기능은 많지만 UI가 올드한 느낌 | 모던하고 직관적 |
| 핵심 기능 | 타일 레이어, 오브젝트 레이어, 커스텀 프로퍼티. 플러그인 풍부 | 오토 타일링(킬러 피처), 엔티티 시스템, 월드 뷰 |
| Phaser 호환 | 네이티브 지원. `load.tilemapTiledJSON()` | 별도 파서 필요 |
| 커뮤니티 | 매우 큼 | 성장 중이지만 아직 작음 |

**결론**: LDtk의 오토 타일링은 매력적이지만, Phaser와의 네이티브 호환이 Tiled에만 있다. 파서를 직접 만드는 비용 대비 Tiled가 합리적이다.

## 이 프로젝트에서 고른 것

**선택: Phaser.js (v3.90.0, 현재 사용 중) + Tiled**

이유:
1. **타일맵 네이티브 지원**: Tiled JSON을 한 줄로 로드하고 레이어별 충돌/인터랙션 설정이 가능하다. PixiJS였으면 이 모든 걸 직접 만들어야 한다.
2. **물리엔진 내장**: Arcade Physics로 벽 충돌, NPC 접근 감지 등을 간단히 처리할 수 있다. 별도 라이브러리 연동 불필요.
3. **학습곡선 대비 생산성**: 1인 개발(백엔드 주력)에서 게임 엔진 자체를 빌드하는 건 비현실적이다. Phaser는 "게임 만들기"에 필요한 것이 다 있다.
4. **커뮤니티와 레퍼런스**: 문제가 생기면 검색으로 해결할 수 있다. Kaplay이나 Excalibur는 이 부분이 취약하다.
5. **이미 통합 완료**: Next.js + Phaser + Zustand 브릿지가 이미 작동하고 있다. 엔진을 바꾸는 건 큰 비용이다.

PixiJS를 고르지 않은 이유:
- Gather Town처럼 PixiJS 위에 자체 프레임워크를 구축할 인력이 없다.
- 타일맵 로더, 물리엔진, 카메라 시스템을 직접 만들어야 하는데, 이건 백엔드 개발자가 혼자 할 일이 아니다.
- 순수 렌더링 성능은 PixiJS가 2배 빠르지만, 이 프로젝트에서 동시에 수천 개 스프라이트를 렌더링할 일은 없다. 마을 하나에 수십~수백 타일 + 캐릭터 몇십 명이면 Phaser로 충분하다.

## 핵심 개념 정리

### Phaser의 구조

```text
Game
└── Scene (VillageScene, InventoryScene 등)
    ├── preload()  — 에셋 로드
    ├── create()   — 오브젝트 생성, 이벤트 바인딩
    └── update()   — 매 프레임 실행 (60fps)

Scene 안에서:
├── Tilemap     — Tiled에서 만든 맵 데이터
├── Sprite      — 캐릭터, NPC, 아이템 (텍스처 + 물리 바디)
├── Camera      — 뷰포트. 캐릭터 따라 스크롤
├── Physics     — Arcade(AABB 충돌) 또는 Matter.js(복잡한 물리)
└── Input       — 키보드, 마우스, 터치 입력 처리
```

### Phaser + Next.js 통합 패턴

이 프로젝트에서 사용하는 패턴은 "분리된 세계" 방식이다:

```text
┌─────────────────────────────────────┐
│  Next.js (React)                    │
│  ├── ChatOverlay (채팅 UI)          │
│  ├── Inventory (인벤토리 UI)        │  ← React가 잘하는 영역: 폼, 리스트, 모달
│  └── LoginPrompt (인증)             │
│                                     │
│  ┌───────────────────────────────┐  │
│  │  Phaser Game (absolute)       │  │  ← Phaser가 잘하는 영역: 스프라이트, 물리, 타일맵
│  │  ├── VillageScene             │  │
│  │  └── Canvas (WebGL)           │  │
│  └───────────────────────────────┘  │
│                                     │
│  Zustand Store (브릿지)             │  ← 두 세계를 연결하는 다리
└─────────────────────────────────────┘
```

핵심은 **Phaser와 React가 각자 잘하는 일을 하고, Zustand로만 소통하는 것**이다.

### Phaser 4 출시 (2026-04-10)

Phaser v4.0.0 "Caladan"이 2026년 4월 10일에 정식 출시되었다. 주요 변경사항:
- WebGL 렌더러 완전 재작성 (노드 기반 아키텍처)
- 모듈러 구조 — 필요한 기능만 포함해 번들 크기 감소
- WebGPU 지원
- 데이터 지향 설계로 대량 스프라이트 성능 향상

현재 프로젝트는 Phaser 3.90.0을 사용 중이다. Phaser 4로의 마이그레이션은 안정화 이후 검토하면 된다.

## 실전에서 주의할 점

- **키보드 입력 충돌**: Phaser가 키보드 이벤트를 캡처하면 React의 input 필드에 타이핑이 안 된다. `document.activeElement` 감지로 해결. (참고: [26-phaser-html-keyboard-focus-conflict](/docs/learning/26-phaser-html-keyboard-focus-conflict.md))
- **메모리 누수**: Scene 전환 시 이벤트 리스너 정리 필수. `scene.events.on()`은 `scene.events.off()`로 정리.
- **HMR 충돌**: Next.js HMR이 Phaser Game 인스턴스를 중복 생성할 수 있다. 인스턴스 중복 방지 로직 필요.
- **번들 크기**: ~1.2MB. 코드 스플리팅으로 게임 씬 진입 시 동적 로드 권장.

## 나중에 돌아보면

- **Phaser가 틀린 선택이라고 느끼는 시점**: 3D 요소가 필요해지는 순간. "마을을 아이소메트릭 3D로 바꾸고 싶다"면 Three.js + R3F 또는 Babylon.js를 검토.
- **PixiJS가 더 나았을 시점**: 프론트엔드 팀이 2명 이상으로 늘어나서 자체 게임 프레임워크를 빌드할 여력이 생겼을 때.
- **스케일 변화 시**: 한 마을에 100명 이상 동시 접속 시 PixiJS로 렌더링 교체 또는 뷰포트 컬링 최적화 필요.
- **Phaser 4 전환 시점**: 생태계(플러그인, 타일맵 지원, 커뮤니티 자료)가 충분히 성숙하면 마이그레이션 고려.
