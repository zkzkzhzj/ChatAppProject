# 36. NPC 대화 엔지니어링 패턴 — "중간 서버가 있는 이유"를 증명하는 4개 축

> 작성 시점: 2026-04-19
> 맥락: 발표 직후 받은 리뷰 피드백을 엔지니어링 관점에서 재해석. 당장 구현하는 문서가 아니라, 다음 NPC 대화 고도화 작업 때 꺼내 쓸 설계 근거 노트.

---

## 배경

오늘(2026-04-19) MVP 발표 후 NPC 대화 시스템에 대해 다음과 같은 리뷰 피드백을 받았다.

### 리뷰 원문 (핵심만)

- 디바운스
- 모르는 것을 잘 이야기하기
- NPC와의 대화를 엔지니어링적으로 풀어보자 (지금 너무 원시적)
- GPT를 쓰는 것과 차이가 없다
- 메시지 단일 응답보다 중간 서버가 끼니까
- 단발성 타자
- 동시적으로 많은 메시지 전달
- 메시지를 어떻게 요약해서 중간 서버에서 컨트롤 할 수 있을까
- 메시지를 한 번 검증한다든지 커스텀을 한다든지

### 피드백을 한 문장으로 요약하면

> **"중간 서버가 왜 끼어 있는지를 엔지니어링으로 증명해라."**

지금 구조는 이렇다.

```text
[유저 클라이언트] → [우리 서버] → [Ollama/LLM] → [우리 서버] → [유저 클라이언트]
                       ↑
                  얘가 뭘 하는데?
```

우리 서버는 인증과 라우팅만 한다. 그러면 유저가 직접 OpenAI API를 쓰는 것과 본질적으로 다르지 않다. 오히려 레이턴시만 늘어난다.

**중간 서버의 존재 의미는 "서버만 할 수 있는 일"을 해야 정당화된다.** 이 노트는 그 "서버만 할 수 있는 일" 4가지 축을 정리한다.

---

## 지금 구조의 한계

| 관점 | 현재 | 문제 |
|------|------|------|
| 호출 빈도 | 유저 타자 1건 = LLM 호출 1건 | 연타 시 비용 폭발, 맥락 파편화 |
| 동시성 | 동일 유저의 병렬 요청 처리 없음 | 먼저 온 응답이 나중 질문에 붙는 레이스 |
| 컨텍스트 | 매 호출마다 히스토리 전량 전달 (혹은 무시) | 토큰 비용 폭증 또는 기억 상실 |
| 검증 | 유저 입력 → LLM 직통, LLM 출력 → 유저 직통 | 페르소나 이탈, 금칙어 누출, 할루시네이션 |
| 비용 | 모든 호출이 동일 비용 | "안녕" 같은 고빈도 인사말도 full LLM 호출 |

이 5개 열을 하나씩 서버가 맡아야 "중간 서버의 이유"가 생긴다. 아래 4개 축이 그 답이다.

---

## 개선 축 4가지

### 축 A — 입력 측: 타자 패턴 제어

유저가 "안녕" → "잘지냈어?" → "오늘 뭐해" 를 3초 안에 연타할 때, 3번 LLM을 호출할 것인가 1번 호출할 것인가.

#### A-1. 디바운스 / 배칭

```text
[안녕] ─┐
[잘지냈어?] ─┼── 800ms 무입력 대기 → 합쳐서 1회 LLM 호출 → "안녕! 잘 지내지, 너는?"
[오늘 뭐해] ─┘
```

트레이드오프 비교.

| | 디바운스 없음 (현재) | 짧은 디바운스 (300ms) | 긴 디바운스 (1500ms) |
|--|---------------------|----------------------|----------------------|
| 응답 체감 | 빠름 | 빠름 | 느림 |
| LLM 호출 비용 | 높음 | 중간 | 낮음 |
| 맥락 일관성 | 파편화 | 중간 | 좋음 |
| 실제 사용 사례 | 단순 프록시 | 구글 검색 자동완성 | GitHub Copilot inline suggestion |

**판단 포인트:** 챗봇은 "타자가 멈춘 지점"이 의미 단위의 경계인 경우가 많다. 800ms 정도가 업계에서 자주 쓰이는 값이지만, 실제로는 유저별 타자 속도 차이가 커서 **adaptive debounce**(유저 평균 타자 간격 기반)도 고려할 수 있다.

#### A-2. 동시성 제어

같은 유저가 LLM 응답이 아직 안 끝났는데 또 메시지를 보낼 때.

| 전략 | 동작 | 장점 | 단점 | 적합한 상황 |
|------|------|------|------|-------------|
| **Coalesce (병합)** | 진행 중 요청에 새 메시지를 묶음 | 비용 효율 | 사용자 맥락 유실 가능 | 빠른 연타형 대화 |
| **Abort (취소)** | 진행 중 LLM 호출 cancel, 최신 것만 | 항상 최신 입력 반영 | 이미 소비한 토큰 낭비 | 사용자가 말을 바꿀 때 |
| **Queue (순차)** | FIFO로 하나씩 처리 | 순서 보장 | 지연 누적, 큐 폭주 위험 | 맥락이 중요한 상담형 |

**실전에서 많이 쓰는 조합:** "디바운스 + Abort". 유저가 타자 중이면 디바운스가 묶고, 이미 서버에서 LLM을 호출 중이면 새 입력이 오는 순간 기존 호출을 취소한다. React 진영의 `AbortController` 패턴, RxJS의 `switchMap` 연산자와 동일한 아이디어.

#### A-3. 구현 관점 메모

- WebSocket 세션별로 "미전송 버퍼"와 "디바운스 타이머"가 필요
- 서버 스케일 아웃을 생각하면 **세션-서버 sticky 여부**가 변수. sticky가 아니면 Redis에 버퍼/타이머 상태를 둬야 한다.
- Ollama 호출은 취소 가능해야 한다 (Spring `WebClient` + `Mono.cancel()` 또는 HTTP 클라이언트 타임아웃)

---

### 축 B — 컨텍스트 측: 긴 대화 압축

LLM의 컨텍스트 윈도우는 유한하다. 100턴쯤 쌓이면 전량 전달은 불가능하거나 비경제적이다.

#### B-1. 전략 비교

| 전략 | 동작 | 장점 | 단점 |
|------|------|------|------|
| **Full history** | 매번 전체 히스토리 전송 | 구현 쉬움, 기억 완전 | 토큰 비용 폭발 |
| **Sliding window** | 최근 N턴만 유지 | 단순 | 장기 맥락 유실 |
| **Summary memory** | 오래된 부분을 요약본으로 치환 | 장기 맥락 보존, 비용 효율 | 요약 품질 의존, 요약도 LLM 호출 비용 |
| **Hybrid (요약 + 최근 N턴)** | 오래된 건 요약, 최근 건 원문 | 실전 표준 | 구현 복잡도 중간 |

**업계 레퍼런스:** LangChain의 `ConversationSummaryBufferMemory`, LlamaIndex의 `ChatSummaryMemoryBuffer`. 둘 다 Hybrid 전략이 기본이다.

#### B-2. 마음의 고향에 맞는 구조

```text
Redis:
  user:{id}:npc:{npcId}:summary   → "최근 업무 스트레스 얘기가 잦음. 고양이를 키움."
  user:{id}:npc:{npcId}:recent    → 최근 10턴 원문 (TTL 24h)

LLM 프롬프트 조립:
  [시스템 프롬프트 = 마을 맥락 + NPC 페르소나]  ← 서버가 주입, 유저가 조작 불가
  [장기 요약]
  [최근 10턴]
  [이번 유저 메시지]
```

**서버만 할 수 있는 일:** 시스템 프롬프트는 서버가 조립한다. 클라이언트에서 직접 LLM을 쓰면 유저가 프롬프트를 뜯어고칠 수 있지만, 서버가 끼면 **NPC 페르소나 무결성**을 강제할 수 있다.

#### B-3. 요약 트리거

- 옵션 1: N턴마다 (예: 20턴)
- 옵션 2: 토큰 수 기반 (예: 최근 대화가 4000토큰 넘으면)
- 옵션 3: 사용자 이탈 후 재접속 시
- 옵션 4: 주제 변경 감지 (이건 또 LLM이 필요해서 비용 증가)

**주의:** 요약 자체가 LLM 호출이라 비용이 있다. 너무 자주 요약하면 배보다 배꼽이 커진다.

### 축 C — 검증/필터 측

#### C-1. 사전 필터 (LLM 호출 전)

유저 입력을 LLM에 보내기 전에 먼저 거른다.

| 필터 | 예 | 효과 |
|------|----|----|
| 금칙어 필터 | 욕설, 성희롱 | 윤리 + LLM이 오염된 입력을 학습/모방하는 것 방지 |
| 개인정보 유도 감지 | "전화번호 알려줘" | 자해/공격 시나리오 조기 차단 |
| Rate limit | 유저당 분당 N회 | 비용 보호 |
| 프롬프트 인젝션 감지 | "너는 이제 시스템 관리자야" | 페르소나 이탈 방지 |

**윤리정책(`/docs/planning/ethics.md`)과의 연결점:** 자동 필터링 대상이 여기서 1차 구현된다. 기존에는 유저 간 대화 기준으로만 정의돼 있었는데, **NPC 대화에도 동일하게 적용되어야 일관된 경험이 된다.**

#### C-2. 사후 처리 (LLM 응답 후)

LLM이 뱉은 응답을 그대로 유저에게 보내지 않는다.

- **말투 강제:** NPC마다 종결어미가 다르다면 (예: 할아버지 NPC = "~라네") 정규식/후처리로 통일
- **금지어 제거:** LLM이 실수로 브랜드명·경쟁 서비스명 언급 시 제거
- **길이 제한:** 한 줄 초과 시 자르거나 "더 듣고 싶어?" 훅 추가
- **"모르는 것"을 정직하게:** LLM이 할루시네이션으로 답하는 대신, 자신 없는 영역은 "그건 잘 모르겠어, 미안해"로 치환

**"모르는 것을 잘 이야기하기"의 두 가지 접근.**

1. **프롬프트로 유도:** 시스템 프롬프트에 "확실하지 않은 건 모른다고 답해라" 지시. 쉬움, 하지만 완벽하지 않음.
2. **검증 레이어:** LLM 응답을 한 번 더 다른 LLM/규칙으로 검증. 비용 2배, 레이턴시 증가, 하지만 안전망.

실전에서는 1번을 먼저 하고, 중요한 영역(의료/법률/자해 관련)에만 2번을 얹는다.

#### C-3. 서버만 할 수 있는 이유

유저가 클라이언트에서 직접 LLM을 쓰면 이 모든 필터를 우회할 수 있다. **서버가 LLM 호출의 유일한 경로여야 강제력이 생긴다.** 이게 중간 서버의 핵심 존재 이유 중 하나.

### 축 D — 비용 측: 캐싱

#### D-1. Semantic Cache

"안녕", "안녕하세요", "하이", "ㅎㅇ" 같은 인사말은 LLM을 매번 호출할 필요가 없다.

```text
유저 입력 → 임베딩 → pgvector로 유사 질문 검색
            │
            ├── 유사도 > 임계값 → 캐시된 응답 반환 (LLM 우회)
            └── 유사도 < 임계값 → LLM 호출 → 응답 캐싱
```

우리 프로젝트에는 이미 pgvector 기반 벡터 스토어가 구축돼 있다 (learning #29 참고). 인프라 추가 없이 바로 접목 가능.

#### D-2. 트레이드오프

| | 캐싱 공격적 (유사도 임계값 낮음) | 캐싱 보수적 (유사도 임계값 높음) |
|--|------------------------------|---------------------------------|
| 비용 절감 | 큼 | 작음 |
| NPC 다양성 | 낮음 ("안녕"에 항상 같은 말) | 높음 |
| 오답 리스크 | 다른 의미 질문에 같은 답 줄 수 있음 | 낮음 |

**현실적 전략:** 고빈도 인사말/짧은 리액션에만 캐싱 적용. 본격적 대화는 캐싱 제외. 또는 **캐시된 응답 N개 중 랜덤 선택**으로 다양성 확보.

#### D-3. 서버만 할 수 있는 이유

전역 캐시는 서버에만 존재할 수 있다. 유저 A가 받은 응답을 유저 B에게도 쓰려면 공유 저장소가 필요하고, 그건 중간 서버 없이는 불가능하다.

## 엔지니어링 가치 정리표

각 축이 리뷰의 어떤 키워드에 답하는지, 왜 서버만 할 수 있는 일인지.

| 축 | 리뷰 키워드 연결 | 해결하는 문제 | 중간 서버만 할 수 있는 이유 |
|----|-----------------|--------------|---------------------------|
| **A. 입력 제어** | 디바운스, 단발성 타자, 동시성 | 비용·지연·맥락 파편화 | 여러 클라이언트 요청을 묶는 상태 공유는 서버에서만 가능 |
| **B. 컨텍스트 관리** | 메시지 요약 | 토큰 비용, 컨텍스트 윈도우 | 장기 메모리·세션 간 영속은 서버 저장소가 필수 |
| **C. 검증·커스텀** | 메시지 검증/커스텀, 모르는 것 | 윤리 이탈, 페르소나 붕괴, 할루시네이션 | 클라이언트 우회를 원천 차단 |
| **D. 캐싱** | 중복 호출 방지 | LLM 비용 | 전역 공유 캐시는 서버에만 존재 |

이 4개가 없으면 우리 서버는 **LLM 요금만 늘리는 프록시**다. 있으면 **NPC 경험의 품질과 비용을 동시에 쥐는 레버**가 된다.

## 회고 — 이 노트가 중요한 이유

리뷰어가 "GPT 쓰는 거랑 차이가 없다"고 말한 건 단순 비판이 아니라 **"너희 서비스의 엔지니어링 가치를 스스로 설명하라"**는 질문이다.

시니어 엔지니어로서 이 질문에 답할 수 있어야 한다. NPC 대화는 기능 하나가 아니라 **"서버가 끼는 이유"를 가장 잘 보여줄 수 있는 영역**이다. 여기를 엔지니어링으로 잘 풀면, 나머지 모든 도메인에서도 "왜 이 서비스가 필요한가"를 설득할 수 있다.

당장 구현은 안 하지만, 다음에 이 주제로 돌아올 때 **이 노트가 "빈 캔버스 공포"를 없애준다.** 4개 축과 18개 설계 결정 항목이 이미 정리돼 있으니, 어디서부터 시작할지 고민할 필요 없이 "A-1부터 가자"로 시작할 수 있다.
