## 0. 이 문서가 왜 있는가 — 메타 메시지

이 노트는 **"결정 기록 휘발 방지"** 를 위해 작성됐다.

세션 중 사용자가 직접 한 말:

> "내가 뭘 선택했고 왜 이걸 선택했는지가 안 남는다."

CLAUDE.md §4 Critical Rule #7 의 정확한 의도가 이거다. 트레이드오프 논의가 발생한 그 응답 안에서 학습 노트를 남기지 않으면, 일주일 뒤 같은 질문이 다시 나왔을 때 **이전 결정을 발견할 단서가 없다.** "나중에 정리하지" 는 "안 한다" 와 같다.

이 노트는 단일 주제가 아니라, 2026-04-25 ~ 04-26 세션에서 사용자가 한 단계씩 의문을 제기하면서 자연스럽게 도출된 **6개 결정의 묶음** 이다. 각 결정은 서로 의존한다 — 1번을 빼면 2번의 의미가 사라진다. 그래서 한 노트로 묶었다.

> 실시간 인프라(`ws-redis` 트랙)와 별개의 도메인 결정이지만, "동시접속이 늘면 어떻게 견디나" 라는 질문이 두 영역을 동시에 건드렸기 때문에 이 노트의 주소가 `ws-redis` 트랙 번호 (46) 에 할당됐다. 트랙 매핑은 [RESERVED.md](./RESERVED.md) 참조.

## 1. 배경 — 어떤 질문에서 시작됐나

부하 테스트 ([learning/43](./43-load-test-breaking-point-story.md)) 에서 VU 200 plateau 가 한계였다. 사용자가 자연스럽게 다음 질문을 던졌다:

> "동시접속이 1000명이면 어떻게 하나? 채널을 미리 10개 열어두자?"

이 질문 하나에서 6개의 서로 얽힌 결정이 도출됐다. 순서대로:

| 결정 # | 질문 | 결정 |
|------|------|-----|
| 1 | 동시접속 한계 도달 시 어떻게 분산? | **Hard Cap** (현 단계). 다중화는 보류 |
| 2 | 같은 마을 내 1000명을 어떻게 견디나 — 채널을 나눠야 하나? | **서버 분리 (수평 확장)**. 채널은 그대로 |
| 3 | "서버 나누기" 의 단위가 뭔가? 물리 서버? | **JVM 프로세스 단위** |
| 4 | 수평 확장 옵션 (같은 EC2 컨테이너 2개 vs EC2 별도) 미리 정해야 하나? | **결정 안 함**. Step 6 부하 측정 후 데이터 보고 |
| 5 | ZEP 처럼 유저가 마을을 직접 만드는 모델? | **보류**. B안 완주 후 별도 트랙 |
| 6 | 마을을 미리 10개 만들어두면? | **사전 생성 X** (빈 마을 역설) |

이 노트는 이 6개를 하나씩, "왜 그렇게 골랐고 다른 건 왜 안 골랐는가" 의 시각으로 정리한다.

## 2. 결정 1 — 동시접속 한계 도달 시: Hard Cap vs 채널 샤딩 vs 마을 다중화

### 2.1 비교

| | A. 동적 채널 샤딩 | B. 정적 사전 생성 | C. 마을 다중화 (ZEP 패턴) | D. Hard Cap |
|--|------------------|------------------|------------------------|------------|
| 핵심 개념 | 인원 기준으로 한 마을을 자동 분할/통합. "채널 1, 채널 2..." | 마을을 처음부터 10개 만들어둠. 유저가 골라 입장 | 1번 마을 만석 → 운영자가 2번 마을 신설 (ZEP 인원 분산 패턴) | 1번 마을 + 상한 도달 시 "지금은 입장 불가, 잠시 후" 안내 |
| 장점 | 유저 인지 부하 0 (자동) · 게임 채널링의 이론적 정점 | 운영 단순 · 컨셉별 마을 가능 | 인원 분산 · 사람이 결정해서 통제 가능 | 구현 0줄 · 도메인 정합성 100% |
| 단점 | **분할 기준 모호** · 메시지 정합성 어려움 · 통합 시점 결정 어려움 · 친구 관계 깨짐 · 디버깅 지옥 | **빈 마을 역설** (§7 결정 6 참조) · NPC 비용 N배 | 운영 인력 필요 · 마을 수 늘면 NPC 비용 N배 · "메인 마을" 인식 고착 | 만석 시 유저 이탈 (서비스가 거절하는 모양새) |
| 적합한 상황 | MMO RPG (메이플, WoW) — **동질적인 게이머**, 채널 선택이 정서적 비용 X | 컨셉이 명확히 분리되는 서비스 (디스코드 카테고리 사전 정의) | 운영 인력 충분 + 유저 감당 가능한 규모 (ZEP) | MVP, 실유저 수십 명 단계 |
| 실제 사례 | 메이플스토리 채널 1~30, 리니지W 서버, MMO instance | Slack workspace · Discord category | ZEP, Gather.town 이벤트 공간 | 대부분의 MVP 서비스 (탈출 시점이 곧 다음 단계) |

### 2.2 선택: D. Hard Cap (현 단계). C는 B안 완주 후 단계적 도입 검토

**근거:**

1. **빈 마을 역설** — 사전 생성하면 "사람 없는 마을 8개" 가 첫인상이다. 도메인 핵심 가치 ("대화가 그리운 사람을 위한 안식처") 정면 훼손. 자세한 분석은 §7.
2. **NPC 비용** — 마을/채널마다 NPC 두면 LLM 호출 N배. 현재 [learning/28](./28-llm-model-selection-and-production-strategy.md) 의 이중 모델 전략은 "마을 1개 + NPC 2~3명" 전제로 비용 예측됨.
3. **운영 복잡도** — 동적 샤딩 (옵션 A) 은 분할 기준을 정해도 "VU 50명 모인 마을을 25/25 로 자동 분할" 시점에 친구 관계가 깨진다. "친구가 다른 채널로 갔는데?" 같은 UX 문제. 통합 시점도 불명확.
4. **우리 규모** — 부하 테스트 [learning/43](./43-load-test-breaking-point-story.md) 에서 VU 200 plateau 가 한계였다. 1000명 시나리오는 "사용자 10만+" 단계의 이야기. 그 단계에선 도메인도 다른 모습일 가능성 큼.
5. **도메인 핵심 가치와의 충돌** — 게이머 정서 ("채널 1번 사람 적네, 5번 가자") 와 안식처 정서 ("어디서 위로받지") 는 다르다. 채널 선택 자체가 추가 인지 비용이고, "내 마을" 이라는 소속감을 약화시킨다.

### 2.3 게임에선 왜 동적 샤딩이 통하나 — 시야 확장

메이플스토리·리니지W·WoW 같은 MMO 가 채널/서버를 미리 잘게 나누는 게 자연스러운 이유:

- **유저 정서가 동질적** — 다 게이머. 채널 선택이 "사냥터 혼잡도 보고 결정" 이라는 게임 메카닉 자체.
- **"빈 채널 = 사냥터 비어있어서 좋음"** — 우리와 정반대 정서.
- **친구 관계가 약함** — 게임 진행상 같은 채널일 필요가 적다 (파티는 채널 합류로 해결).

우리 도메인에선:

- **유저가 "대화 상대" 를 찾으러 옴.** 빈 공간은 곧 실패.
- **친구 관계가 강함.** 같은 마을에 있는 게 핵심.
- **선택 자체가 부담.** "어디 가야 사람 있지?" 라는 질문은 안식처 컨셉과 충돌.

이 차이를 무시하고 게임 패턴을 베끼면 도메인이 깨진다. 사용자가 직관적으로 채널 분할을 떠올린 건 **"동시접속 분산" 이라는 문제 인식 자체는 옳았기 때문**. 다만 분할 단위가 도메인 (채널) 이 아니라 인프라 (서버) 였을 뿐이다 — 이건 결정 2에서 다룬다.

## 3. 결정 2 — 같은 마을 1000명을 어떻게: 채널 분리 vs 서버 분리

### 3.1 사용자 의문

> "메인 마을 1000명 동시접속이면 어떻게 견디나? 채널 안 나누면 한 서버가 다 받는 거 아닌가?"

이 질문이 본질적으로 옳다. 그 답이 **"채널이 아니라 서버를 나눈다"** 라는 것.

### 3.2 비교

| | A. 채널 분리 (도메인 분할) | B. 서버 분리 (인프라 분할) |
|--|--------------------------|---------------------------|
| 분할 단위 | "마을 1번 채널 1, 채널 2..." 도메인 개념 | JVM 프로세스 N개. 유저 입장에선 "마을 1번" 그대로 |
| 유저 UX | 채널 선택 강제 (인지 부하) | 변화 없음 (서버는 유저가 모름) |
| 메시지 라우팅 | 채널별로 격리. 채널 1 메시지가 채널 2에 안 감 | 같은 마을이면 어느 서버에 붙었든 모두에게 도달 |
| 친구 관계 | 다른 채널이면 만남 깨짐 | 영향 없음 |
| 구현 난이도 | "채널" 이라는 도메인 개념 추가 필요 | 코드 변경 0. Redis Pub/Sub 추상화가 이미 처리 |
| 한계 | 채널당 인원이 단일 서버 한계 도달하면 또 분할? | Redis 단일 인스턴스 한계 (수백만 메시지/s) 까지 선형 확장 |

### 3.3 선택: B. 서버 분리

**근거:**

1. **Redis Pub/Sub 의 본질적 가치가 여기서 나온다.** 채널 = 도메인 개념, 서버 = 인프라 개념. Redis Pub/Sub 이 둘을 **직교 (orthogonal)** 로 만들어준다.
   - 한 채널 (`chat:room:1`) 을 N대 서버가 동시에 SUBSCRIBE.
   - 서버 1은 자기 250명에게만 fan-out, 서버 2는 자기 250명에게만, ...
   - Redis 입장에선 publish 1번에 N개 서버에 메시지 1번씩 전달 (n=서버수, k=세션수에 무관).
2. **사용자 인지 부하 0** — "어느 서버에 접속하셨나요?" 같은 질문 X. 로드밸런서가 알아서 분배.
3. **빅테크 표준** — Slack · Discord · LINE LIVE 모두 같은 패턴 ([learning/45 §1.5](./45-websocket-redis-pubsub-redesign.md#15-line-live-의-3층-actor--redis-bridge--우리-버전)).
4. **사용자 직관 보정** — "100명씩 나눠야 한다" 는 직관 자체는 옳다. 단지 **분리 단위가 채널이 아니라 서버였을 뿐**.

### 3.4 그림으로

```text
[채널 분리 — 도메인 분할]                [서버 분리 — 인프라 분할]
유저 입장에선 5개 채널 보임            유저 입장에선 마을 1개 그대로

마을 1, 채널 1 ─── 서버 A              ┌── 서버 A (250명)
마을 1, 채널 2 ─── 서버 A              │
마을 1, 채널 3 ─── 서버 A              마을 1 ──┤── 서버 B (250명) ──┐
마을 1, 채널 4 ─── 서버 A              │                            │
마을 1, 채널 5 ─── 서버 A              └── 서버 C (250명)            │
                                                                   ▼
※ 채널 간 메시지 격리                                          Redis Pub/Sub
※ 친구가 다른 채널이면 못 만남                                 chat:room:1
                                                                   ▲
                                                                   │
※ 한 메시지 → 모든 서버 → 그 서버가 자기 250명에게 fan-out ─────┘
```

### 3.5 빅테크 사례 — 더 깊이

[Slack Engineering — Real-time Messaging](https://slack.engineering/real-time-messaging/) 에서 명시적으로 다룬다:

- workspace = 도메인 개념 (변하지 않음)
- "channel server" = 인프라 단위 (스케일 따라 N대)
- 유저는 한 channel server 에 sticky session 으로 붙고, message 는 다른 channel server 가 받았어도 "subscription routing layer" 가 모든 channel server 에 fan-out.

**이 프로젝트에서 Slack 의 "subscription routing layer" 자리를 채우는 게 Redis Pub/Sub.**

LINE LIVE ([learning/45 §1.5](./45-websocket-redis-pubsub-redesign.md#15-line-live-의-3층-actor--redis-bridge--우리-버전)) 도 동일 — Akka actor 가 로컬 fan-out, Redis 가 서버 간 bridge.

[12-factor App](https://12factor.net/processes) 의 "Stateless processes + Backing services" 원칙도 같은 맥락. 프로세스는 무상태, 상태는 외부 backing service (Redis) 에 위임 → 프로세스를 N개로 늘려도 동일 동작.

## 4. 결정 3 — "서버" 단위는 무엇인가

### 4.1 사용자 의문

> "서버 나누기란 물리 서버 단위?"

이 질문이 정확하다. 인프라 용어가 4개 층위라 한 번 정리할 가치가 있다.

### 4.2 4개 층위

```text
┌──────────────────────────────────────────────────────────┐
│ 물리 서버 (PM, Physical Machine)                          │
│  AWS 데이터센터의 진짜 H/W 박스                            │
│  ┌────────────────────────────────────────────────────┐  │
│  │ VM / EC2 인스턴스                                    │  │
│  │  하이퍼바이저로 가상화된 OS 단위                      │  │
│  │  ┌──────────────────────────────────────────────┐  │  │
│  │  │ 컨테이너 (Docker)                             │  │  │
│  │  │  cgroups + namespace 격리. OS 공유            │  │  │
│  │  │  ┌────────────────────────────────────────┐  │  │  │
│  │  │  │ JVM 프로세스                            │  │  │  │
│  │  │  │  TCP 연결을 가지는 단위                  │  │  │  │
│  │  │  │  Redis 입장에선 "subscriber 1개"        │  │  │  │
│  │  │  └────────────────────────────────────────┘  │  │  │
│  │  └──────────────────────────────────────────────┘  │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 4.3 선택: JVM 프로세스 단위

**근거:**

- Redis Pub/Sub 관점에서 의미 있는 단위는 **TCP 연결을 가지는 단위** = JVM.
- 물리 서버에 EC2 가 몇 대 올라가 있든, EC2 안에 컨테이너가 몇 개 있든, 컨테이너 안에 JVM 이 몇 개 있든 — Redis 는 무관.
- 따라서 "서버 N대로 확장" 의 N 은 **JVM 프로세스 수**. 그게 EC2 1대에 컨테이너 2개로 구현되든 EC2 2대로 구현되든 코드는 동일.

### 4.4 실전 시사점

- "스케일 아웃 1대 추가" 라고 할 때 그게 무슨 의미인지 명확히 — JVM 프로세스 1개 추가.
- 헬스체크·모니터링도 JVM 단위로 한다 (EC2 단위가 아님). EC2 한 대가 살아있어도 JVM 이 죽으면 그 인스턴스는 빠진 것.
- ALB target group 등록 단위도 JVM 의 listening port. EC2 IP 단위가 아닌 (host:port) 단위.

## 5. 결정 4 — 수평 확장 옵션 단계: 미리 정할까

### 5.1 사용자 의문

> "같은 EC2 안에 컨테이너 2개 vs EC2 2대 vs Scale Up. 어느 걸로 갈지 미리 정해야 하나?"

### 5.2 옵션 비교

| | A. Scale Up | B. 같은 EC2 컨테이너 2개 | C. EC2 별도 |
|--|-------------|----------------------|-------------|
| 개념 | 단일 JVM RAM/CPU 증가. EC2 instance type 키움 | 같은 EC2 에 JVM 2개 (컨테이너 2개) | EC2 자체를 2대로 |
| Redis 필요? | ❌ 불필요 (단일 JVM 인메모리) | ✅ 필요 | ✅ 필요 |
| 비용 | EC2 instance 사이즈 가격 (지수 곡선) | EC2 1대 그대로 + 컨테이너 추가 (사실상 무료) | EC2 2대 (선형) |
| 자원 격리 | N/A | ❌ 같은 cgroup 안에서 분리 정도 | ✅ 완전 격리 |
| 장애 격리 | ❌ JVM 죽으면 끝 | ❌ EC2 죽으면 둘 다 죽음 | ✅ 한 대 죽어도 다른 대로 트래픽 |
| 빠른 시작 | ✅ 가장 단순 | ✅ docker-compose scale 한 줄 | ⚠️ EC2 launch + 보안그룹 + ALB target group 등록 |
| 운영 복잡도 | 최저 | 중 | 고 |
| 가용성 (AZ 분리) | 단일 AZ | 단일 AZ | 멀티 AZ 가능 |

### 5.3 선택: 결정 안 함 — Step 6 부하 측정 후 데이터 보고 결정

**근거:**

1. **코드는 어느 쪽이든 동일.** Redis Pub/Sub 추상화 덕분에 B든 C든 코드 차이 0. 인프라 토폴로지 차이만 있을 뿐.
2. **사전 결정의 가치가 없다.** "Step 6 부하 재측정" 에서 단일 JVM 한계점이 어디인지 측정하고, 거기서 시작하는 게 합리적.
3. **B 와 C 사이의 트레이드오프는 본질적으로 "장애 격리 비용" vs "시작 속도".** 이건 운영 단계의 의사결정이지 설계 단계의 결정이 아니다.

### 5.4 단, 큰 방향성은 잡아둠

- Scale Up (A) 은 **선택 안 함**. 단일 JVM 인메모리 구조의 한계는 부하 테스트에서 이미 봤다 (VU 200 plateau, dispatch thread 단일성).
- B 와 C 사이는 **부하 측정 후 결정**. MVP 단계는 B (같은 EC2, 컨테이너 2개) 로 시작 가능성 높음 — 빠르고 비용 0. 가용성이 요구되는 시점에 C 로 이행.
- B → C 이행 비용은 "EC2 launch + ALB target group 등록" 정도. 코드 변경 없음.

### 5.5 시야 확장 — Scale Up 의 한계가 왜 본질적인가

JVM 1개의 한계:

- **GC 일시 정지** — heap 이 커질수록 Full GC 시간 증가. 8GB heap 이면 G1GC 도 100ms 단위 정지. WebSocket 메시지 처리에는 치명적.
- **단일 dispatch thread (현 구조)** — Simple Broker 가 그랬듯 단일 스레드 병목. 멀티스레드로 짠다 해도 NIC/socket buffer 단일 큐 한계.
- **장애 격리 0** — JVM crash 가 곧 서비스 전면 중단.

이래서 빅테크가 다 Scale Out (수평 확장) 으로 가는 것. Scale Up 은 RDB 같은 **상태 보유 시스템** 에서나 의미 있다 (Aurora 가 그래서 이상). 무상태 애플리케이션 서버는 무조건 수평이 정답.

## 6. 결정 5 — ZEP 처럼 유저가 마을을 직접 만드는 모델?

### 6.1 사용자 의문

> "젭처럼 사람이 방을 생성하고 입장하는 건?"

### 6.2 비교

| | A. 운영자 1개 마을만 (현재) | B. 신뢰 등급 게이트 (중간 안) | C. 누구나 마을 생성 (ZEP 풀 패턴) |
|--|------------------------|----------------------------|--------------------------------|
| 마을 수 | 1개 | 수십 ~ 수백 | 무제한 |
| 빈 마을 위험 | 없음 (1개라 항상 사람 있음) | 낮음 (만든 사람 + 친구가 첫 입주민) | 매우 높음 (만들고 안 들어옴) |
| 도메인 정합성 | ✅ "안식처" 명확 | ⚠️ 마을의 무게감 약화 시작 | ❌ 안식처 → 단순 채팅방 무리 |
| 컨텐츠 다양성 | 낮음 | 중 (자생적 분화) | 높음 |
| NPC 비용 | 1마을 분 | 마을당 NPC 둘 건지 정책 결정 필요 | 폭증 (마을당 NPC) |
| 운영 복잡도 | 낮음 | 중 (신고/삭제/심사 정책) | 높음 (악용 방지·부적절한 마을 등) |
| 우리 도메인 가치 | ✅ 정합 | ⚠️ 위험 | ❌ 충돌 |

### 6.3 선택: 보류 — B안 완주 후 별도 트랙으로 검토

**근거 (왜 매력적인가):**

- ✅ 빈 마을 문제 해결: 만든 사람 + 친구가 첫 입주민 → 자연스러운 시작.
- ✅ 컨텐츠 다양성: 유저 자생적으로 컨셉 마을 생성 ("새벽 감성 마을", "수험생 위로 마을").
- ✅ 유저 lock-in: 자기가 만든 마을은 애착 강함.

**근거 (왜 보류인가):**

- ❌ "안식처" 정체성 흐려짐. 마을의 무게감 소멸 → "그냥 채팅방 모임".
- ❌ NPC 비용 폭증. 마을마다 NPC 두면 LLM 호출 N배. 안 두면 빈 마을 양산.
- ❌ 운영 복잡도 폭증. 부적절한 마을 신고/삭제 정책 필요.
- ❌ 도메인 핵심 가치 ("대화가 그리운 사람을 위한 안식처") 와 직접 충돌. 안식처는 **운영자가 책임지는 공간** 이지 유저가 마음대로 만드는 공간이 아니다.

**중간 안 (B) 의 가능성:**

- 신뢰 등급 도달한 유저만 신설 가능 + 마을 컨셉 태그 + 운영자 검토.
- 마을 수 상한 제어 가능. 컨텐츠 다양성 일부 확보.
- 단, 이것도 **B안 (raw WS + Redis Pub/Sub) 완주 후** 의 다음 단계.

### 6.4 채팅 인프라에는 영향 없음

이게 핵심 분리 포인트.

- ws-redis 트랙 (`chat:room:{id}`) 은 마을 1개든 N개든 매핑 동일.
- 마을 운영 모델은 **도메인 결정** 이고, 채팅 인프라는 **그 결정을 그대로 매핑** 만 하면 됨.
- 따라서 ws-redis 트랙 진행에 마을 운영 모델은 dependency 가 없다 → 별도 트랙으로 분리해도 안전.

### 6.5 ZEP 패턴이 ZEP 에선 통하는 이유 — 시야 확장

ZEP (zep.us) 은 한국 메타버스 플랫폼. 누구나 공간 (=마을 격) 을 만들고 친구를 초대한다.

ZEP 에선 통하는 이유:

- **이벤트성/일회성 공간** 이 주된 사용 패턴 (회사 워크숍, 동아리 모임). "안식처" 가 아니라 "모임 장소".
- **비어있는 공간이 정상** — 행사 끝나면 비어도 됨. 우리 도메인과 정반대.
- **운영자가 ZEP 사 본인이 아님** — 공간 생성한 유저가 운영. 책임 분산.
- **NPC 가 핵심 컨셉이 아님** — 사람만 모이는 게 컨셉.

우리 도메인은:

- **상시성 공간** — 언제든 가면 사람이 있어야 한다.
- **비어있으면 실패** — 도메인 가치 훼손.
- **운영자 = 우리 본인** — 마을 품질 책임을 우리가 짐.
- **NPC 가 핵심** — 빈 마을 방지 + 안식처 정서 형성에 필수.

**같은 패턴인데 도메인이 다르면 결과가 정반대.** 그래서 패턴만 보고 베끼면 안 된다.

비슷한 사례:

- **Gather.town** — ZEP 비슷한 미국 서비스. 공간 무제한 생성. 같은 이유로 우리 도메인엔 부적합.
- **동물의 숲: 모여봐요 (닌텐도)** — 무인도가 1인용 안식처. 우리 도메인에 가장 가까운 정서. 단, 친구 최대 7명 초대(섬 동시 인원 최대 8) 만 가능 — **마을 무제한 생성과 안식처 정서가 양립 가능한가?** 라는 질문에 닌텐도가 내린 답은 "1인 + 소수 친구 초대" 였다는 것.

## 7. 결정 6 — 빈 마을 역설: 마을을 미리 N개 만들어둘까

### 7.1 사용자 의문

> "마을을 미리 10개 열어두자?"

이건 결정 1의 옵션 B (정적 사전 생성) 가 왜 안 되는지에 대한 별도 분석. 같은 이유가 ZEP 패턴 (결정 5) 의 위험성과도 연결되기 때문에 분리해서 명시.

### 7.2 빈 마을 역설이란

**짧게:** "선택지를 늘릴수록 빈 마을이 늘고, 빈 마을이 보일수록 유저가 안 들어와서 더 빈다."

```text
초기 상태:
  마을 10개 사전 생성 → 유저 100명 입장
  로드밸런싱 X · 유저는 1번 마을부터 들어옴
  결과: 1번 마을 95명, 2~10번 마을 5명 분산 (편중)

신규 유저 첫 인상:
  "1번 마을 = 사람 많음. 2~10번 마을 = 사람 거의 없음 (= 죽은 마을)"
  → 1번 마을로 더 쏠림
  → 2~10번 마을은 영영 비어있음

결과:
  9개 유령 도시 + 1개 만석 마을
  "안식처가 아니라 버려진 메타버스" 첫인상 → 회복 불가
```

### 7.3 비교

| | A. 사전 생성 (10개) | B. Hard Cap (1개 + 만석 시 안내) |
|--|------------------|-----------------------------|
| 신규 유저 인지 부하 | "10개 중 어디?" 추측 비용 | 0 (들어오면 바로 마을) |
| 첫 인상 | 빈 마을 다수 → 죽은 서비스 인식 | 사람 있는 마을 → 살아있는 서비스 |
| 도메인 정합성 | ❌ 안식처 가치 훼손 | ✅ 정합 |
| 운영 (NPC 등) | 마을 10개 분 | 마을 1개 분 |
| 한계 | 만석 시점이 와도 9개 유령 그대로 | 만석 시점에 다음 결정 필요 (결정 1·5 가 그 답) |

### 7.4 선택: B. 사전 생성 X

**근거:**

1. **첫 인상은 회복 불가** — "사람 없는 마을 = 죽은 마을" 인식이 한 번 박히면 SEO/입소문에서 못 회복. ZEP/Gather.town/동물의 숲 신작 모두 같은 함정 경험.
2. **도메인 핵심 가치 훼손** — "안식처에 갔는데 사람이 없다" 는 단 한 번의 경험이 모든 마케팅 비용을 무력화.
3. **NPC 비용** — 마을 10개면 NPC도 10배. 빈 마을일수록 NPC 만 있는 이상한 광경.
4. **현재 단계에 맞지 않음** — MVP 실유저 < 10명. 마을 1개도 채우기 빠듯한 단계에 10개 생성은 무의미.

### 7.5 사전 생성이 통하는 도메인 — 시야 확장

빈 마을 역설이 안 통하는 (즉 사전 생성이 정답인) 도메인:

- **Discord 카테고리/채널** — 워크스페이스 운영자가 사전 정의. 유저가 만든 게 아니라 "지정된 토픽". 빈 채널이어도 죽은 인상 X (아직 토픽이 활성화 안 됐을 뿐).
- **Slack channels** — 동상.
- **회사 메신저 채널** — 부서별 사전 분리. 빈 채널 = 활동이 없는 부서. 죽은 인상 아님.

**공통점**: "조직 구조" 에 매핑된 채널은 사전 생성해도 자연스럽다. 사람이 모이는 "장소" 로서의 마을은 그렇지 않다.

이래서 ZEP 도 사전 생성된 공간은 거의 없고 (랜딩 공간 정도) 유저 생성 모델로 간 것. 우리는 ZEP 의 유저 생성 모델도 도메인 충돌로 안 가니까, 결국 "마을 1개 + Hard Cap" 이 도메인적으로 가장 정합한 선택.

## 9. 실전에서 주의할 점

- **"동시접속이 늘면 어떻게 하나" 라는 질문이 또 나올 것.** 이 노트로 답한다 — 답은 "Hard Cap (현재) → 서버 분리 (확장 시) → 마을 다중화 (도메인 검증 후)".
- **"채널 나누자" 는 직관에 다시 끌리지 말 것.** 그건 게임 도메인의 정답이지 안식처 도메인의 정답이 아니다.
- **Step 6 부하 재측정 결과가 나오면** 결정 4 (B vs C) 를 그때 결정. 측정 데이터 없이 미리 결정하지 말 것.
- **유저 수 증가 시그널** 이 나오면 결정 5 (ZEP 패턴) 를 다시 검토. 다만 "B안 완주 후" 라는 전제는 유지 — 인프라 검증 끝나기 전에 도메인 모델 흔들면 동시 변경으로 추적 불가.
- **NPC 비용** 은 마을 수에 비례한다. 마을 다중화 검토 시 LLM 비용 예측 ([learning/28](./28-llm-model-selection-and-production-strategy.md)) 재계산 필수.

| 단일 JVM 부하 측정 결과 50명에서도 한계 | 결정 4 (B vs C) | C (EC2 별도) 로 직행. AZ 분리까지 |
| Redis 단일 인스턴스 CPU 50% 도달 | 결정 2 (서버 분리) 의 한계 | Redis 클러스터 또는 NATS ([learning/45 §6](./45-websocket-redis-pubsub-redesign.md#6-나중에-돌아보면)) |
| "안식처" 가 컨셉으로 작동하지 않음 (유저 피드백) | 도메인 가치 자체 재정의 | 이 노트 전체가 무효. 새 노트에서 새 도메인 가치 정의부터 |

### 스케일이 바뀌면 어떤 결정 조합이 적합한가

| 단계 | 마을 모델 | 서버 모델 | 비고 |
|-----|---------|---------|-----|
| MVP (~50명) | Hard Cap, 마을 1 | 단일 JVM | 현재 |
| 초기 성장 (~500명) | Hard Cap, 마을 1 | JVM 2~3 (옵션 B/C) | Step 6 측정 후 |
| 본격 성장 (~5000명) | 마을 다중화 (운영 신설, 컨셉별) | JVM 5~10 + Redis 클러스터 검토 | 결정 1 재검토 |
| 대규모 (~50000명) | 신뢰 등급 + 유저 신설 (중간 안 B) | NATS 또는 자체 라우터 | 결정 5 재검토 |
| 초대규모 (~Discord) | 별도 도메인 모델 | 전용 스택 (Erlang/Elixir) | 거의 다른 서비스 |
