# 67. Wiki 폐지 권고를 철회한 이유 — 노화의 원인을 잘못 짚었다

> 작성 시점: 2026-04-30
> 트랙: `harness-spec-driven` (Issue #46) — 메타·도구 트랙
> 출발점: AI Native 하네스 분석 중 "wiki 14페이지 4-15 이후 lint 미실행" 신호
> 산출물: [`wiki-policy.md`](../conventions/wiki-policy.md) — 활용 강화 4종 자동화

---

## 0. 이 노트는 어떻게 읽어야 하나

이 노트는 **"내가 잘못된 권고를 했고, 사용자가 그걸 짚었고, 다시 분석해서 결론을 뒤집었다"** 의 기록이다.

기술 결정의 기록이 아니라 **분석 오류 패턴의 기록.** 같은 함정에 다시 빠지지 않게.

---

## 1. 처음 권고 — "wiki 축소 또는 폐지"

세션 분석에서 wiki 의 신호:

- `docs/wiki/` 14 페이지 (frontend / communication / identity / infra / village)
- `docs/wiki/log.md` 마지막 lint: **2026-04-15** (분석 시점 기준 15일 전)
- 운영 흐름: lint 후 "재실행 권장" 상태로 멈춰 있음

이 신호 + AI Native 트렌드 (learning 시점 기록 / spec 요구사항 / 코드 현재 / handover 진행) 4축이 wiki 와 역할 중복으로 보임 → **"wiki 폐지 또는 축소" 권고**.

추천 옵션 3개 제시 (A 폐지 / B 축소 / C 유지). 권장은 B (축소).

---

## 2. 사용자가 던진 한 줄

> "wiki는 왜 처분해? 지금 의미가 없다는거야?"
> "wiki를 쓸 환경을 고민하는게 좋을거같아"
> "카파시 LLM Wiki가 핫했거든"

핵심:

- wiki 는 **의도적으로 도입한 자산**. 노화 신호 하나만 보고 폐지 판단 X
- 결단 전에 **활용 환경** 을 먼저 고민하라
- 카파시 LLM Wiki 패턴은 그 자체로 가치

→ **이 한 줄로 권고 전체를 다시 봐야 했다.**

---

## 3. 재분석 — 노화의 원인은 wiki 가 아니다

### 처음 분석의 틀린 가정

> "wiki 가 묵었다 = wiki 의 가치가 떨어진다"

### 실제 인과

```text
wiki 갱신 동선이 없다
  ↓
트랙 종료 시 "어떤 wiki 가 영향받았나" 자동 탐지 X
  ↓
갱신 시점이 임의·수동
  ↓
잊힌다 → 노화
  ↓
(잘못된 추론) "wiki 자체가 가치 없다"
```

→ **노화의 원인은 wiki 자체가 아니라 갱신 동선 부재.** 동선만 만들면 wiki 는 살아난다.

### 4축 시제 재정렬 — wiki 와 다른 자산이 진짜 중복인가?

| 자산 | 시제 | 답하는 질문 |
|------|------|-----------|
| **Wiki** (`docs/wiki/`) | **현재형** | 이 시스템이 *지금* 어떻게 작동하는가? |
| Spec (`docs/specs/features/`) | 미래형 | 이 트랙으로 무엇을 달성하나? |
| Track (`docs/handover/track-*.md`) | 진행형 | 지금 어디까지 했나? |
| Learning (`docs/learning/`) | 과거형 | 왜 이 결정을 했나? |
| 코드 | 현재형 (상세) | 정확한 동작 |

→ wiki 와 코드는 둘 다 현재형이지만 **다른 추상 수준.** 코드는 상세, wiki 는 그림.

> **"새 인입자 (또는 새 LLM 세션) 가 코드를 읽지 않고 시스템 그림을 잡는 유일한 자산."**

learning / spec 으로 대체 불가능. wiki 가 없으면 새 세션은 매번 14개 패키지를 코드부터 읽어야 한다.

## 4. 결론을 뒤집은 결정 — 활용 강화 4종 자동화

[`wiki-policy.md`](../conventions/wiki-policy.md) §2 의 4종:

1. **`/track-end` 의 wiki 영향 분석** (P3) — 트랙이 건드린 도메인 자동 매핑 → 영향 페이지 체크리스트 → 머지 PR 안에 wiki 커밋 묶기
2. **`wiki-lint` 주간 cron** (P4) — 매주 월요일 09:00 KST. 회수된 dependency-tracker 슬롯 사용
3. **`/spec-new` 의 wiki 링크 의무 입력** (P3) — spec.references 에 wiki 페이지 link 비우면 강제 보임
4. **SessionStart hook 의 wiki 가시화** (P4) — `wiki/INDEX.md` last-modified 출력. 30일 이상 묵으면 ⚠️

→ **갱신을 강제하지 않고 "잊을 수 없게" 만드는 동선.** 사용자에게 강제 X, 가시화 O.

## 5. 분석 오류의 패턴 — 무엇이 잘못됐나

| 오류 단계 | 무엇을 했나 | 무엇을 했어야 했나 |
|---------|------------|------------------|
| 신호 해석 | "lint 미실행 → wiki 가치 X" | "lint 미실행 → 갱신 동선 부재" 로 인과 재배치 |
| 옵션 설계 | 폐지 / 축소 / 유지 (자산 처분 관점) | 노화 원인 별 처방 (동선 / 자산 / 전제 가정) |
| 사용자 의도 검증 | 제안 후 사용자 답 대기 | **"이 자산을 도입한 의도" 를 먼저 묻기** (카파시 패턴 의도) |
| 추천 강도 | "권장: B (축소)" — 강한 추천 | "옵션 비교만, 추천 보류" — 자산 처분은 신중 |

### 한 줄 룰 (이번부터 따른다)

> **"의도적으로 도입된 자산" 을 노화 신호 하나로 처분하지 마라. 노화 원인을 먼저 인과로 분해하라.**

## 6. 카파시 LLM Wiki 패턴 — 시야 확장

Andrej Karpathy 가 "The hottest new programming language is English" 발언 이후, LLM 이 코드를 직접 읽기보다 **잘 정리된 markdown 문서로 시스템을 이해** 하는 게 효율적이라는 흐름이 강해졌다.

### 핵심 전제

- 코드 읽기는 토큰 비싸고 컨텍스트 부담 큼
- 잘 정리된 wiki 1 페이지가 같은 정보를 1/10 토큰으로 전달
- 새 세션 진입 시 코드보다 wiki 먼저 읽으면 컨텍스트 효율 ↑

### 적용 사례

- Karpathy 자신의 nanogpt 등 미니 프로젝트가 wiki-style README 강조
- HuggingFace / LangChain 등 OSS 가 docs-first 흐름
- AI Native 개발에서 "LLM 친화적 markdown 베이스" 가 production grade 자산으로 인정받음

### 마음의 고향에서 wiki 가 의미 있는 이유

- 솔로 개발 + Claude Code 메인 사용자 → **새 LLM 세션이 매주 시작.** 매번 코드부터 읽으면 토큰·시간 폭주
- 14 페이지 (frontend / communication / identity / infra / village) 가 도메인 그림을 1 화면에 압축
- 갱신 자동화만 갖추면 wiki 는 카파시 패턴의 **production 적용 사례** 가 됨

## 7. 나중에 돌아보면 — 이 결정이 틀릴 수 있는 조건

- 갱신 자동화 4종이 도입됐는데도 wiki 가 6개월간 안 갱신되면 → 폐지 검토 재개
- 새 LLM 세션이 wiki 를 안 읽고 코드만 읽는 흐름이 측정되면 → wiki 진입점 강화 필요
- 14 페이지가 50+ 페이지로 늘어나면 → wiki 자체의 인덱스 부담 → 카테고리 재구성 검토

## 8. 더 공부할 거리

- **Diataxis 프레임워크** (tutorials / how-to / reference / explanation) — 우리 wiki 가 어느 카테고리인가? "reference" 가 가장 가까움. 다른 카테고리도 별도 디렉토리로 둘 가치 있나?
- **Cursor / Continue / Aider** 등 다른 AI 코딩 도구의 docs 활용 패턴 — wiki 가 LLM 컨텍스트로 자동 주입되는 메커니즘이 있는가?
- **C4 Model** (Context / Container / Component / Code) 다이어그램 — wiki 가 C4 의 어느 레벨까지 그려야 효율인가?
- **Doc-as-code** 흐름 (docs 가 코드와 같은 PR/리뷰 사이클) — 우리 정책 이미 그렇게 가고 있음. 강화 여지?
