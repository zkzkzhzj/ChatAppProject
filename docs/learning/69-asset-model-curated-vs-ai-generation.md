## 0. TL;DR — 무엇이 걸려 있는가

마을의 캐릭터·옷·가구 같은 시각 에셋의 **공급 모델**을 정한다. 한 번 정하면 비즈니스 모델, 운영 인력, 인프라 비용, 시각 일관성, 차별점이 전부 따라온다. **잘못된 길을 가면 되돌리는 비용이 매우 크다** — 이미 발급된 에셋, 이미 결제된 인벤토리, 학습된 사용자 기대치가 묶여 있기 때문.

세 옵션을 두고 따진다:

| | A. 큐레이션 | B. AI 생성 | C. 하이브리드 |
|--|------------|-----------|--------------|
| 비유 | ZEP/Gather | DALL·E + Phaser | 옷가게 + 자수 기계 |
| 비즈니스 모델 | 검증됨 (가챠·구매) | 미지의 영역 | 가능하지만 quota 정책 추가 |
| 차별점 | 약함 | 강함 | 중간 |
| 운영 부담 | 디자이너 필요 | API 비용·필터·저작권 | 두 시스템 운영 |
| 마음의 고향 가치 정합 | 보통 | 강함 (개인화) | 강함 (개인화 + 일관성) |

## 1. 배경 — 왜 지금 결정해야 하는가

### 1.1 마음의 고향의 가치 매핑

`docs/planning/project-overview.md` §2 핵심 가치 두 줄:

- **"대화가 그리운 사람을 위한 안식처"** — 들어오는 순간 "여기는 편한 곳"이라는 감각
- **공간 꾸미기 = 유저 유입의 핵심 전략** (§3.1) — "내 안식처를 만드는 행위"

이 두 줄이 에셋 모델 결정에 직접 들어온다. 안식처 감각은 **시각 일관성**(들어왔을 때 톤이 흔들리지 않음)과 **개인화**(내 공간이 정말 "내" 공간이라는 느낌) 사이의 미묘한 지점에서 생긴다. 어느 한쪽으로 치우치면 가치가 깨진다:

- 시각 일관성만 추구 → ZEP 처럼 "범용 가상 공간" 이 되어 차별점 없음
- 개인화만 추구 → 무절제한 비주얼로 "안식처" 가 아니라 "혼란스러운 마을" 이 됨

### 1.2 결정의 무게

이 결정이 영향을 주는 영역:

- **수익 모델**: 광고 보상 → 포인트 → 아이템 구매 흐름이 (A) 인벤토리 모델인지 (B) 디자인 슬롯 모델인지에 따라 완전히 다름
- **인프라 비용**: (B)/(C) 는 LLM API 비용이 월간 고정 지출로 들어옴
- **운영 인력**: (A) 는 디자이너, (B) 는 프롬프트·필터·저작권 운영, (C) 는 둘 다
- **사용자 기대치**: 한 번 "AI 가 옷 만들어준다" 라는 경험을 주면 되돌리기 어려움

### 1.3 결정 전 사용자 자문 — 4축 트리거

스펙 결정 형식(decisions/triggers)으로 사용자가 먼저 답해야 할 4질문:

1. **수익 모델 의도** — 가챠/구매 베이스인가, 광고-포인트 베이스인가?
   → 가챠는 (A) 인벤토리 모델과 결합력이 강함. quota 모델은 (B)/(C) 와 자연.
2. **출시 타임라인** — 1~2개월 안에 MVP 띄우고 싶은가, 6개월 이상 가능한가?
   → AI 파이프라인 (프롬프트 튜닝·필터·저작권) 학습 비용은 최소 1개월. (A) 가 빠름.
3. **운영 인력** — 혼자 운영하는가, 디자이너 합류 가능성이 있는가?
   → 혼자면 (C) 의 베이스 큐레이션 부담이 가장 작음.
4. **"안식처" 가치를 어디서 표현할 것인가** — 시각인가, 대화/NPC 인가, 마을 분위기인가?
   → 시각 비중이 크면 (B)/(C). 다른 영역이면 (A) 로 충분.

> 이 4질문에 대한 답이 없으면 옵션 비교는 무의미하다. 본 노트는 "어느 옵션이 옳다" 가 아니라 "각 답에 따라 어느 옵션이 적합한가" 의 매트릭스를 제공한다.

## 2. 세 옵션 — 깊은 비교

### 2.1 옵션 A — 에셋 구매·입기 (전통적 큐레이션)

**컨셉**: 디자이너가 만든 에셋 카탈로그에서 유저가 골라서 구매·착용·배치한다. ZEP, Gather.town, 모바일 가챠 게임의 표준 모델.

**장점**:
- **검증된 비즈니스 모델** — 가챠, 구매, 시즌 패스 등 수십년 데이터가 있음
- **시각 일관성 보장** — 같은 디자이너 손에서 나오므로 톤이 안 흔들림
- **기술 단순** — 정적 에셋 + DB 인벤토리 + Phaser 스프라이트 로딩. 인프라 부담 거의 없음
- **큐레이션 쉬움** — "이번 주 신상" 같은 마케팅 훅이 자연스러움
- **저작권 명확** — 디자이너와 계약 끝나면 에셋 소유권이 명확함

**단점**:
- **안식처 가치와 약한 연결** — 카카오톡 이모티콘 사면서 "안식처" 라고 느끼지 않음
- **디자인 작업 부담** — 매주 신상 못 내면 콘텐츠 고갈, 유저 이탈
- **차별점 부족** — ZEP·Gather·모바일 게임 다 하는 모델. "왜 마음의 고향이어야 하는가" 답하기 어려움
- **개인화 한계** — "내 공간" 이라고 해도 결국 1만명이 같은 카탈로그에서 고른 결과. 진짜 "내" 가 아님

**숨은 빈틈**:
- **에셋 양산 못 따라가면 콘텐츠 고갈** — 디자이너 1명으로 주 10~20개 신상 내는 게 한계. 유저가 1만명만 넘어도 "다 본 것" 이 됨
- 이걸 막으려면 디자이너 추가 채용, 외주, UGC 중 하나로 가야 함 — 다 비용

**적합한 상황**:
- 출시 타임라인이 짧고 (1~2개월 MVP)
- 가챠/구매 비즈니스 모델 의도가 명확하며
- 시각 차별점은 약해도 다른 영역(NPC, 대화 톤)에서 차별화할 자신이 있을 때

**실제 사례**:
- **ZEP** — 범용 2D 공간, 큐레이션 카탈로그, 기업 행사용으로 안착
- **Gather.town** — 비슷한 모델, 업무용 가상 오피스
- **모바일 가챠** (페그오·우마무스메·블루아카이브) — "수집의 재미" 가 핵심. 캐릭터 한정·등급·합성 등의 메커니즘으로 인벤토리 모델을 비즈니스 엔진으로 승화

### 2.2 옵션 B — AI 생성 (Google AI Studio / OpenAI API / Stability)

**컨셉**: 유저가 "분홍색 체크무늬 원피스" 같은 프롬프트를 입력하면 AI 가 즉석에서 옷을 생성한다. 캐릭터·옷·가구·공간 모두 AI 가 만든다.

**장점**:
- **강력한 차별점** — "AI 가 내 안식처를 같이 만들어준다" 는 메시지가 마음의 고향 가치와 자연스럽게 연결
- **무한 콘텐츠** — 콘텐츠 고갈 없음. 유저 상상력이 곧 카탈로그
- **개인화 극대** — 진짜 "내" 공간. ZEP 1만명 중 1명이 아니라 1명만의 공간

**단점**:
- **API 비용 폭발** (2026-05 시점, [IntuitionLabs · LaoZhang AI](https://intuitionlabs.ai/articles/ai-image-generation-pricing-google-openai)):
  - DALL·E 3: $0.016~$0.02/image
  - Imagen 4 Fast/Standard/Ultra: $0.02 / $0.04 / $0.06
  - Gemini 2.5 Flash Image: $0.039
  - **무료 사용자 100명 × 하루 10장 = 월 $480~$1,800** (Imagen 4 Ultra 기준 $1,800)
  - 사용자 1,000명 시 $4,800~$18,000 — **광고 보상 모델의 마진 구조와 충돌**
- **품질 일관성** — Phaser 2D 도트/픽셀 톤과 AI 생성물(주로 사실적 일러스트 톤) 불일치. 마을 전체 톤이 흔들림
- **레이턴시** — 5~30초. "옷 사면 바로 입는다" 의 즉각성이 깨짐. 게임 흐름 끊김
- **인벤토리 모델 붕괴** — "구매한 아이템이 내 보관함에 있다" 가 안 됨. 무한 생성이라 재고 개념 자체가 무의미. 가챠/구매 모델 못 씀
- **저작권/책임** — AI 가 학습 데이터의 IP 를 침해한 결과물을 생성하면 책임은 누가? 운영자가 짐
- **컨텐츠 필터** — 유저가 "성적/혐오 이미지" 프롬프트 시 차단 필요. 운영 부담

**숨은 빈틈**:
- **cross-user 시각 일관성** — A 유저가 만든 옷과 B 유저가 만든 옷이 같이 마을에 있을 때 톤이 너무 다르면 마을 자체가 "안식처" 느낌이 깨짐
- **quota 정책 자체가 설계 부담** — 무료 N장/일, 유료 무제한 같은 구조를 처음부터 짜야 함. quota 도달 시 UX, 결제 흐름까지 다 설계 필요

**적합한 상황**:
- 차별점이 가장 중요하고 (다른 서비스와의 격차)
- 광고 모델이 아니라 **구독 모델** ("월 $5 로 무제한 AI 생성") 의도가 있으며
- 시각 톤 불일치를 감수할 수 있는 (또는 모든 에셋이 AI 라 일관됨) 디자인 방향일 때

**실제 사례**:
- **Character.AI** — 캐릭터 자체는 AI 생성하지만, 비주얼은 정적 (대화에 집중)
- **Scenario.gg / Layer / Tripo3D** — 게임 에셋 AI 생성 도구, 인디 개발자가 **개발 단계에서** 쓰는 도구. **유저가 런타임에 생성** 하는 게 아님
- **Google Project Genie 3** (2026-01 공개, [Cybernews 리뷰](https://cybernews.com/ai-tools/google-project-genie-review/)) — 텍스트로 3D 환경 생성. 아직 실시간 멀티플레이 게임 런타임 사용은 실험 단계

> 산업 사례에서 주목할 점: **유저가 런타임에 AI 로 에셋을 생성하는 라이브 서비스 사례는 매우 드물다.** 거의 모든 사례는 "개발자가 AI 로 만든 에셋을 정적으로 배포" 하는 방향. 이게 옵션 B 의 신호다.

### 2.3 옵션 C — 하이브리드 (베이스 큐레이션 + AI 악센트)

**컨셉**: 시각의 뼈대는 큐레이션으로 고정하고, 유저가 변형할 수 있는 좁은 영역만 AI 가 생성한다.

**구조**:
- **베이스(큐레이션)**: 캐릭터 골격, 마을 타일, 기본 옷 셋, 가구 골격
- **AI 영역**: 옷 패턴/색, 액세서리, 인테리어 일부, 감정 표현 스티커
- **비용 통제**: 일일 quota + "디자인 슬롯" 저장 (예: 무료 3슬롯, 광고 시청 +1)
- **스타일 강제**: 시스템 프롬프트로 "Phaser 2D 픽셀 톤", "파스텔 컬러 팔레트" 강제

**장점**:
- **차별점 유지** — "AI 가 옷 패턴 만들어준다" 라는 메시지 유지
- **시각 일관성 유지** — 베이스가 고정이므로 마을 톤은 안 흔들림
- **비용 통제 가능** — quota 가 명시적이므로 비용 상한선 예측 가능
- **인벤토리 모델 유지** — "디자인 슬롯" 이 곧 인벤토리 → 가챠/구매 메커니즘 적용 가능
- **점진 도입 가능** — A 로 시작하고 작은 영역(옷 색·패턴)부터 C 로 확장 가능

**단점**:
- **두 시스템 운영 부담** — 큐레이션 파이프라인 + AI 파이프라인 둘 다 유지
- **베이스/AI 경계 정의 비용** — "어디까지가 AI 영역인가" 를 처음에 잘못 그으면 나중에 재설계 비용 큼
- **시스템 프롬프트 한계** — 스타일 강제가 100% 안 됨. AI 가 종종 톤을 깨는 결과 냄
- **사용자 기대 관리** — "AI 인데 왜 형태는 못 바꾸냐" 같은 기대 충돌

**숨은 빈틈**:
- **AI 영역의 "범위 확대 압박"** — 유저가 "옷 색은 되는데 형태도 바꾸고 싶다" 고 요구. 한 번 양보하면 결국 옵션 B 로 끌려감. **경계 방어 정책이 필수**
- **두 시스템의 시각 톤 차이** — 베이스는 디자이너 톤, AI 영역은 모델 톤. 둘이 안 어울리면 오히려 더 어색

**적합한 상황**:
- 운영 인력이 작고 (혼자~소수)
- MVP 출시 후 **점진 확장** 의도가 있으며
- 가챠/구매 + AI 차별점을 둘 다 살리고 싶을 때
- → **마음의 고향 현재 상황과 가장 가까움 (추정)**

**실제 사례**:
- **Roblox UGC** — 베이스 캐릭터 골격은 고정, 옷·액세서리는 사용자 제작. AI 가 아니라 **사용자가 제작** 이지만 "베이스 + 변형" 구조의 검증된 사례
- **Minecraft 텍스처팩** — 베이스 메시는 고정, 텍스처(그림)만 사용자 변경
- **모바일 게임 캐릭터 커스터마이징** — 골격·얼굴은 고정, 옷·머리색만 변경

## 3. 4축 트리거 × 옵션 매트릭스

사용자가 §1.3 의 4질문에 답하면 다음 매트릭스로 옵션 추천:

| 축 | 답 | 추천 |
|----|----|------|
| 수익 모델 | 광고-포인트-구매 | A 또는 C |
| 수익 모델 | 구독 (월정액) | B 또는 C |
| 출시 타임라인 | 1~2개월 MVP | A |
| 출시 타임라인 | 6개월+ | B 또는 C |
| 운영 인력 | 혼자 | C (또는 작은 A) |
| 운영 인력 | 디자이너 합류 | A 또는 C |
| 안식처 가치 표현 | 시각 비중 큼 | B 또는 C |
| 안식처 가치 표현 | 대화/NPC 비중 큼 | A |

**모든 축이 C 를 가리키는 답이 가장 많을 것으로 추정** (혼자 운영 + 광고 모델 + 시각 차별점 욕심 + MVP 우선). 단, 사용자가 직접 답해야 함.

## 4. PoC 권장 — Step 0 검증

옵션 C 로 가더라도 **바로 풀 구현하지 말고 PoC 로 측정** 하라:

**PoC 설계 (1주)**:
- 범위: 옷 색·패턴만 AI 생성, 그 외는 모두 큐레이션
- 모델: Gemini 2.5 Flash Image ($0.039/장) 또는 Imagen 4 Fast ($0.02/장)
- quota: 무료 3장/일, 광고 시청 +1
- 측정 지표:
  - 비용/유저/일 — 실제 사용 패턴
  - 품질 컴플레인율 — 사용자가 "이상하다" 라고 한 비율
  - 톤 일관성 — 디자이너가 봤을 때 마을이 깨지는가
  - 레이턴시 — 5~30초 대기 시 이탈률
  - quota 도달률 — 무료 quota 가 의미 있게 소진되는가 (도달률 너무 낮으면 차별점 약함, 너무 높으면 비용 폭발)

**PoC 결과로 결정할 것**:
- 옵션 C 그대로 갈지
- 옵션 A 로 후퇴할지 (PoC 가 비용/품질에서 안 됨)
- 옵션 B 로 확장할지 (PoC 가 너무 잘 돼서 더 큰 영역 AI 화)

> 마플 인사이트 §8 — "AI 를 이용해서 잘 운용되는 서비스가 나오기 시작하면 진지하게 고민. 하지만 코드 단위로 신중." PoC 가 그 "코드 단위 신중" 의 실천이다.

## 5. 마음의 고향 가치와의 정합도 — 의견 (참고)

> 결정자는 사용자. 본 섹션은 도메인 가치 매핑을 사람이 다시 따져볼 때 참고용.

**(A) 큐레이션** — "안식처" 가치와의 연결이 약함. 1만명이 같은 카탈로그 = "내" 가 아님. **차별점이 약해서 카카오톡 익명채팅·블라인드와 비교당함**.

**(B) 풀 AI** — "안식처" 가치와 가장 강하게 연결. 그러나 **시각 톤 불일치가 안식처 감각을 깨면 가치 자체가 무너짐**. 이건 마음의 고향에서 가장 위험한 실패 모드.

**(C) 하이브리드** — 안식처 가치와 잘 연결되면서 시각 일관성도 지킬 수 있음. **단, 베이스/AI 경계를 잘 그어야 함**. 잘못 그으면 양쪽 단점만 떠안음.

**핵심 트리거**: 사용자가 "유저가 자기 공간에 들어왔을 때 가장 강하게 느꼈으면 하는 감각" 을 한 줄로 정의해야 함.
- "여기는 내가 직접 만든 곳" → B/C
- "여기는 편하고 따뜻한 곳" → A/C

## 6. 마플 인사이트 정합 (자기 점검)

`memory/marpple_coffee_chat_insights.md` 의 시그널과 본 결정의 정합:

- **§1 트레이드오프 토론** — 본 노트가 그 형식. 옵션 한 개를 칭찬하지 않고 셋 다 단점을 명시
- **§4 "깊게 파고든 흔적"** — 비용 모델·산업 사례·PoC 설계까지 따져본 흔적
- **§7 함수형 본질 — 단점까지 답할 수 있어야 함** — 본 노트는 옵션 B 의 단점을 7개 명시. 면접에서 "AI 도입했어요" 가 아니라 "AI 의 이런 한계까지 검증했고 C 로 후퇴했어요" 가 되도록
- **§8 "AI 끌려다니지 말라"** — 옵션 B 가 가장 위험. C 의 "베이스 큐레이션 + 좁은 AI 영역 + quota" 가 "AI 활용 ✅ 끌려다님 ❌" 의 구체적 형태

> 마플 인사이트가 옵션 B 를 무조건 거르라고 하는 게 아니다. **옵션 B 를 선택하더라도 그 단점·한계를 설명할 수 있는가** 를 점검하라는 것. 본 노트가 그 점검을 거친 결과여야 함.

## 7. 재검토 트리거 — 어떤 신호가 오면 다시 보는가

옵션을 한 번 결정해도 다음 신호가 오면 재검토:

| 신호 | 어느 옵션으로 다시 보는가 |
|------|-------------------------|
| 월 AI API 비용 > 매출 30% | C → A 후퇴 (또는 quota 강화) |
| 사용자 컴플레인 "톤이 깨진다" 비율 > 5% | B → C 후퇴 (또는 스타일 강제 강화) |
| quota 도달률 < 10% | AI 영역 무의미 → A 로 통합 |

| 사용자가 "다 본 것" 이라는 신호 (이탈) | A → C 확장 |
| Phaser 톤과 AI 톤 불일치가 눈에 띔 | C 의 베이스/AI 경계 재정의 |
| 경쟁사가 같은 모델로 진입 | 차별점 재설계 (옵션 무관) |
