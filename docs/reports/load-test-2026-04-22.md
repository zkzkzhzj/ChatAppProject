### 2.4 근본 원인 분석 — JWT 만료 × 토큰 풀 재사용

**타임라인**:

| 시각 | 사건 |
|------|------|
| 15:22 KST | `prepare-tokens.js COUNT=1000` — 1000개 JWT 발급 후 `tokens.json` 저장 |
| 17:00 KST | C-4 시작 |
| 경과 | **1시간 38분** |

**백엔드 설정** (`backend/src/main/resources/application.yml:57`):

```yaml
jwt:
  access-token-expiry-ms: 3600000  # 1시간
```

→ 1000개 토큰 **전부 만료 상태**로 테스트 실행됨.

**실패 메커니즘**:

1. k6 → CONNECT 프레임의 `Authorization: Bearer {expired-jwt}` 전송
2. Spring `StompAuthChannelInterceptor` → `parseTokenPort.parse(token)` → `Optional.empty()`
3. `throw new MessageDeliveryException("Invalid or expired token")`
4. Spring STOMP ERROR 프레임 + WS close 1002 (protocol error) 응답
5. k6 iteration 즉시 종료 → 다음 iteration → 무한 반복

**즉, 부하 테스트가 아니라 "만료 JWT rejection 테스트"가 됨**. Cloudflare·백엔드 자체 문제 아님.

**진단 경로** (블로그 Task 1 서사 재료):

```text
handshake 101 OK 1건 확인 → WS 레이어는 정상
→ CONNECTED 프레임 0건 → STOMP 레이어에서 reject
→ close 1002 protocol error → 서버가 명시적으로 STOMP 프레임을 거부
→ StompAuthChannelInterceptor 코드 분석
→ jwt.access-token-expiry-ms = 3600000 (1h) 발견
→ tokens.json 발급 시각 추적 → 1h 38m 경과 확정
```

### 2.6 C-4 2차 실행 — 실제 breaking point 확정 🎯 (2026-04-22 17:21 KST)

**선행**: 토큰 재발급 완료 (login 1000 / 0 errors / 82.5s). 모든 JWT는 테스트 시작(17:21:30 ±) **이전에** 발급 완료 — 테스트 전 구간에서 만료(1h) 여유 확보.

**커맨드**: C-4 1차와 동일 (`--stage 1m:50 --stage 3m:500 --stage 5m:1000 --stage 2m:0`).

**실행 결과**: 약 6분 7초 경과 시점(**VU 713 도달**) 수동 중단.

#### 에러 시작 좌표

| 시각 | 경과 | 이벤트 |
|------|------|--------|
| 17:21:30 (±) | T+0 | 테스트 시작 |
| **17:22:02** | **T+32s** | **VU 53에서 첫 `handshake failed: status=502`** ← **Breaking point** |
| 17:22~17:27 | T+32s ~ T+6m07s | 실패 지속, VU ramp 따라 계속 증가 |
| 17:27:37 (±) | T+6m07s | k6 강제 종료 (kill switch 발동) |

**이전 실패와의 차이**: C-4 1차는 `status=0` (JWT 만료 → WS close 1002) 였으나 이번은 `status=502/520` (HTTP 레이어 자체에서 원본이 5xx 반환). **WS upgrade 단계도 못 감** — 백엔드가 HTTP 요청조차 처리 못 하는 상태.

#### 에러 유형별 카운트 (~6분간 누적)

| 상태 코드 | 건수 | 의미 |
|-----------|------|------|
| `status=0` | 20,712 | CF/중간 네트워크가 연결 drop (응답 없음) |
| `status=520` | 15,310 | CF가 origin에서 invalid 응답 받음 |
| `status=502` | 24 | CF가 origin에서 5xx 응답 받음 |
| **총** | **36,046** | **handshake 완료 건수 0** |

**STOMP ERROR / close 1002 / close 1006 모두 0건** — WS upgrade 자체에 실패하므로 STOMP 레이어까지 도달 못 함.

#### 백엔드 상태 (kill 직후)

| 엔드포인트 | 응답 |
|-----------|------|
| `/actuator/health` | **504 Gateway Timeout** (60s 대기 후) |
| 여러 차례 재시도 | 지속 504 |

k6 중단 후에도 백엔드가 회복 안 됨. Tomcat 쓰레드 풀 전체가 hanging WS 세션에 묶여있거나, 내부 broker dispatch queue가 폭주 상태로 추정.

#### 병목 원인 후보 (유력도 순, Grafana 17:21~17:27 구간 관찰로 확증 필요)

1. **STOMP Simple Broker single-threaded dispatch** ⭐
   - `org.springframework.messaging.simp.broker.SimpleBrokerMessageHandler`가 모든 SUBSCRIBE/SEND를 단일 worker thread로 처리
   - VU당 초당 2회 position SEND → broker가 subscriber 전원(VU_N명)에게 broadcast
   - VU 50 기준: 초당 약 100 inbound × 50명 fan-out = **5,000 dispatch/s**
   - VU 500이면 250,000 dispatch/s — Simple Broker가 감당 못 함
   - 확증 지표: JVM Threads Live 증가 패턴·MessageHandler thread state
2. **Tomcat worker thread pool 고갈 (기본 200)**
   - WS 업그레이드 + 세션 유지 + inbound 처리에 쓰레드 점유
   - 꽉 차면 HTTP 요청(health check 포함)이 queue에서 대기 → CF 504
   - 확증: Threads Live가 200 근처 평탄화 구간 + HTTP Rate 감소
3. **JVM Heap 압박 (256 MiB 상한)**
   - 세션·MessageHeaders·STOMP 프레임 객체 대량 할당
   - G1 Mixed GC 빈번 → 순간 pause가 서비스 응답 차단
   - 확증: Heap 90%+ 진입 · GC Pause Max 급등 · Old Gen 급성장
4. **JWT 파싱·쓰레드 풀 경합**
   - StompAuthChannelInterceptor가 CONNECT마다 ParseTokenPort 호출 → JWT 파싱 CPU 사용
   - 가능성 낮음 (CPU 관찰 필요)

#### 결론 — Breaking Point 좌표

**약 VU 50 내외에서 백엔드 한계 도달**.

- 1000명은 한참 못 미치는 수준
- Phase 3 설계 단계에서 "스케일아웃 시 외부 broker로 교체 예정"이라 문서화한 [docs/learning/15](../learning/15-websocket-stomp-deep-dive.md), [learning/21](../learning/21-village-public-chat-architecture.md)의 가정이 **실측으로 입증**됨
- Simple Broker는 수십 동시 접속 규모까지가 현실적 한계

#### 2.7.2 단일 원인 아님 — **3중 동시 포화 + GC Death Spiral**

breaking point는 하나의 병목이 아니라 **연쇄 반응**:

```text
① VU ~50 진입 (17:20 KST)
    ↓
② 50개 WS 세션 · Subscription · MessageHeaders · 각 VU당 초당 2 position SEND
    → Simple Broker가 각 SEND를 50명에게 fan-out
    → 초당 5,000 dispatch = MessageHeaders·Message 객체 대량 생성
    ↓
③ Heap 22% → 87% in ≤15초 (사실상 순간 포화)
    ↓
④ G1 Mixed GC 트리거 → Full GC 승격 → Major GC 1.65초 STW 🔴
    ↓
⑤ 1.65초 동안 모든 애플리케이션 쓰레드 정지:
    - Tomcat worker threads → HTTP 요청 수락 불가 → 큐에 쌓임
    - STOMP broker dispatch → 메시지 처리 정지
    - Scheduled Outbox relay → 쿼리 실행 불가
    - /actuator/prometheus scrape → 504 Timeout
    ↓
⑥ GC 끝난 직후 heap 여전히 높음 (해제된 건 일부 Young Gen만)
    + 1.65초 동안 쌓인 대기 요청·메시지가 한꺼번에 들어옴
    ↓
⑦ Tomcat thread pool이 대기 요청을 받으려 급팽창 72 → 180+
    → max-threads 200 접근 → 이후 요청 거부 시작
    ↓
⑧ CPU 70% 도달 — GC thread + 대기 큐 drain + broker dispatch 동시
    ↓
⑨ 또 Heap 꽉 참 → 또 Major GC → 반복 (Death Spiral)
    ↓
⑩ 서버 사실상 정지 상태
```

#### 2.7.3 왜 VU ~50에서 정확히 꺾이는가

- Simple Broker fan-out = **VU² × 2 dispatch/s** (각 VU가 초당 2회 SEND · 전원에게 broadcast)
  - VU 10 → 200/s (C-3.5 문제없음)
  - VU 50 → 5,000/s (broker thread + 객체 생성 rate 임계치)
  - VU 100 → 20,000/s
  - VU 1000 → 2,000,000/s (이론상 계산 불가능 수준)
- Heap 256 MiB로는 5,000 dispatch/s의 객체 churn을 Young GC로 수거 못 함 → Old Gen promotion 폭증 → Major GC 트리거
- Tomcat 200 threads도 50 WS 세션(각자 subscribe + send + receive 3 path) = 150 threads 점유에 + health check + actuator 등 감안하면 여유 없음
- CPU 2 vCPU (t3.medium 추정)로는 Simple Broker의 single-threaded dispatch + GC + 네트워크 I/O 병행 처리 한계

**결론**: 이 인프라 스펙 (256 MiB heap · 2 vCPU · Tomcat 기본)에서 **Simple Broker 기반 STOMP**는 VU 50 내외가 한계. 이는 Phase 3 설계 문서 ([learning/15](../learning/15-websocket-stomp-deep-dive.md), [learning/21](../learning/21-village-public-chat-architecture.md)) 의 "스케일아웃 필요 시 외부 broker 교체" 가정을 **실측으로 입증**한 결과.

#### 2.7.4 Prometheus scrape 공백이 확증하는 것

5개 패널 모두 17:22~17:35 구간 공백. 이건 단순히 "데이터 없음"이 아니라 **서버가 `/actuator/prometheus` 엔드포인트조차 응답 못 할 정도로 마비**되었다는 측정 가능한 증거. 블로그 Task 1에 "모니터링 시스템이 모니터링 대상을 못 따라잡았다" 서사로 사용 가능.

#### 2.7.5 수습 방향 — 3가지 선택지

| 옵션 | 방식 | 예상 효과 | 공수 | 리스크 |
|------|------|---------|-----|-------|
| **A. 단기 패치** — JVM heap `-Xmx 512m` + Tomcat `server.tomcat.threads.max=400` | deploy/.env · docker-compose 수정 · 재배포 | VU ~100~150 까지 상향 예상. Simple Broker 한계는 여전 | 30분 | 낮음. 근본 해결 아님 |
| **B. 근본 해결** — STOMP Simple Broker → Redis Pub/Sub (external) | WebSocketConfig에 `enableStompBrokerRelay` · Redis 인프라 · 설정 마이그레이션 | VU 500~1000+ 확장 가능. 멀티 인스턴스 지원 | 1~2일 | 중. 채팅 아키텍처 재설계 |
| **C. 코드 최적화** — position throttling (서버에서 10Hz → 2Hz 강제) · broker outbound thread pool 증설 | 컨트롤러·broker config 수정 | VU ~100 선까지 개선. 실시간 반응성 일부 저하 | 4시간 | 낮음. UX 트레이드오프 |

권장 조합: **A(즉시) → C(코드 최적화) → 부하 재측정 → 필요 시 B(장기)**.

## A. Appendix — 실행 전 계획 원본

이 섹션은 C-4 착수 **이전**에 작성한 계획서·체크리스트·예상 병목 후보다. 실제 실행 결과는 §2.6~§2.8 참조. 보존 목적(계획 vs 실제 대조)으로만 남긴다.

### A.1 처음 계획한 프로파일

```text
ramping-vus: 0 → 50 (1m) → 500 (3m) → 1000 (5m) → 0 (2m)
총 11분 (gracefulRampDown 30s 포함)
```

실제로는 이 프로파일에서 VU 53 즈음 breaking 발생하여 VU 1000 목표 미도달. 이후 256MB heap에서는 더 높은 VU를 견딜 수 없다고 확인 → 1GB heap 증설(PR #23) 후 §2.8 Plateau Sweep 으로 VU 200까지 재측정.

### A.2 착수 시점 예상 병목 후보

1. **WebSocket Simple Broker** (단일 서버 인메모리 — 유력) ← **실제 주범으로 확인됨**
2. Cassandra dual-write (message + user_message)
3. Outbox `@Scheduled` 1s polling
4. 메시지 카운터 `ConcurrentHashMap`
5. DB connection pool
6. JVM heap (384MB 당시 기준 → GC 빈번 가능) ← **실제 주범 #2로 확인됨** (256MB로 축소된 상태에서 Major GC STW)
7. Cloudflare DDoS 오인 차단 (경로 선택에 따라) ← **발동 안 함**

### A.3 착수 시점 Grafana 감시 대상 6종

| 패널 | 당시 idle 값 | 실행 전 설정 경계선 |
