1. 이 작업은 왜 화면에서 티가 안 나는가

네가 느낀 “동작은 같은데 소스만 바뀐 것 아닌가”라는 감각은 맞다. 이번 단계의 목표는 유저가 보는 채팅 UI, 캐릭터 이동, 타이핑 표시를 바꾸는 것이 아니다. 목표는 같은 기능을 다른 실시간 통신 배관으로 흘릴 수 있는지를 검증 가능한 상태로 만드는 것이다.

화면에서 성공 조건은 “달라 보인다”가 아니다. 성공 조건은 “같아 보이는데 Network 탭의 WebSocket 경로와 Frames가 달라졌다”이다.

기존에는 Spring STOMP와 SimpleBroker가 라우팅과 broadcast를 대신 처리했다. 새 후보 경로는 브라우저가 JSON envelope를 직접 보내고, 백엔드 raw handler가 해석한 뒤, Redis Pub/Sub을 통해 방 단위 fan-out을 수행한다. 즉 사용자는 같은 채팅을 보지만 서버 내부에서는 “STOMP broker에게 맡긴 일”을 “우리 코드와 Redis relay가 명시적으로 수행”하게 된다.

2. 현재 소스 지도

현재 프로젝트는 모놀리식 Spring Boot 애플리케이션 안에 STOMP 경로와 raw V2 경로가 같이 있다. 프론트도 하나의 Next.js 앱 안에서 transport facade가 STOMP/raw를 선택한다. 즉 “MSA가 됐다”가 아니라 모놀리식 안에 두 번째 실시간 transport 후보가 들어왔다가 정확한 표현이다.

운영 기본 STOMP 경로

↓ SockJS/STOMP
backend WebSocketConfig endpoint /ws
↓ @MessageMapping
ChatMessageHandler, PositionHandler, TypingHandler
↓ SimpMessagingTemplate
Spring SimpleBroker /topic, /queue

선택 가능한 raw V2 경로

↓ native WebSocket JSON

↓ ChatWebSocketHandler

3. STOMP 모놀리식 소켓 흐름

기존 운영 경로는 STOMP over WebSocket이다. STOMP는 WebSocket 위에서 동작하는 메시징 프로토콜이다. 클라이언트가 SEND, SUBSCRIBE 같은 STOMP command를 보내면 Spring이 destination을 보고 @MessageMapping 또는 broker topic으로 라우팅한다. 

Browser
  -> SockJS / STOMP Client
  -> /ws
  -> STOMP CONNECT Authorization header
  -> /app/chat/village        -> ChatMessageHandler
  -> /app/village/position    -> PositionHandler
  -> /app/village/typing      -> TypingHandler
  -> SimpMessagingTemplate
  -> /topic/chat/village
  -> /topic/village/positions
  -> /topic/village/typing
  -> Browser subscribers

3.1 설정: WebSocketConfig

WebSocketConfig.java는 STOMP 경로의 출발점이다. @EnableWebSocketMessageBroker를 켜고, /ws endpoint를 등록한다. 클라이언트가 서버로 보내는 메시지는 /app prefix를 사용하고, 서버가 클라이언트에게 broadcast하는 destination은 /topic과 /queue다. 

config.enableSimpleBroker("/topic", "/queue");
config.setApplicationDestinationPrefixes("/app");
registry.addEndpoint("/ws").withSockJS();

여기서 중요한 단어는 SimpleBroker다. 이 broker는 애플리케이션 JVM 안에서 동작한다. 그래서 단일 서버에서는 편하지만 여러 서버로 늘릴 때 같은 topic 구독자를 서버 전체에서 공유하지 않는다.

3.2 채팅: ChatMessageHandler

STOMP 채팅은 /app/chat/village로 들어온다. 서버는 principal이 회원인지 확인하고, SendMessageUseCase로 메시지를 저장한 뒤, SimpMessagingTemplate.convertAndSend로 ChatTopics.VILLAGE_CHAT에 broadcast한다. 

@MessageMapping("/chat/village")
public void handleMessage(StompSendMessageRequest request, Principal principal) {
  // 회원만 채팅 가능
  // SendMessageUseCase.execute(...)
  // messagingTemplate.convertAndSend(ChatTopics.VILLAGE_CHAT, ...)
}

비즈니스 저장 흐름은 application service에 있다. transport가 STOMP든 raw든 “유저 메시지를 저장한다”는 핵심 유스케이스는 SendMessageUseCase가 담당한다. 이것이 헥사고날 구조에서 중요한 분리다.

3.3 위치와 타이핑

위치는 /app/village/position으로 들어와 /topic/village/positions로 나간다. 타이핑은 /app/village/typing으로 들어와 /topic/village/typing으로 나간다. 둘 다 DB 저장 없이 실시간 broadcast에 가깝다.

기능 Inbound Outbound 핸들러
채팅 /app/chat/village /topic/chat/village ChatMessageHandler
위치 /app/village/position /topic/village/positions PositionHandler
타이핑 /app/village/typing /topic/village/typing TypingHandler
메일 알림 서버 내부 이벤트 /user/queue/mail STOMP user destination

3.4 STOMP 경로의 장점과 한계

장점

  • Spring이 destination routing을 제공한다.
  • 구독, 메시지 publish, user queue 개념이 이미 있다.
  • 빠르게 기능을 만들기에 좋다.
  • 메일 알림처럼 사용자별 queue를 쓰기 쉽다.

한계

  • SimpleBroker는 JVM 내부 broker라 스케일아웃 때 서버 간 구독 상태 공유가 어렵다.
  • fan-out 병목이 Spring messaging 계층에 숨어 보일 수 있다.
  • STOMP/SockJS 프레임과 reconnect 정책이 클라이언트 디버깅을 복잡하게 만든다.
  • raw WebSocket보다 프로토콜 오버헤드와 추상화가 크다.

4. raw WebSocket V2 흐름

raw V2는 STOMP 프로토콜을 제거한 후보 경로다. 브라우저가 native WebSocket으로 /ws/v2에 연결하고, JSON envelope를 직접 주고받는다. 

Browser
  -> new WebSocket("ws://localhost:8080/ws/v2?access_token=...")
  -> { "type": "SUBSCRIBE", "roomId": 1 }
  -> { "type": "POSITION", "roomId": 1, "x": 123, "y": 456 }
  -> { "type": "TYPING", "roomId": 1, "typing": true }
  -> { "type": "PUBLISH", "roomId": 1, "body": "hello" }
  -> ChatWebSocketHandler
  -> RoomMessageBus.publish(...)
  -> Redis channel chat:room:1
  -> RoomSubscriptionRegistry fanOut(...)
  -> Browser

4.1 endpoint 등록: WebSocketV2Config

raw V2는 @EnableWebSocket를 추가로 쓰지 않는다. Spring에서 @EnableWebSocket과 @EnableWebSocketMessageBroker를 동시에 켜면 handler mapping 충돌 가능성이 있기 때문이다. 그래서 현재 코드는 WebSocketHttpRequestHandler와 SimpleUrlHandlerMapping으로 /ws/v2를 직접 등록한다. 

WebSocketHttpRequestHandler requestHandler =
    new WebSocketHttpRequestHandler(chatV2HandlerProvider.getObject());

mapping.setUrlMap(Map.of("/ws/v2", requestHandler));
mapping.setOrder(0);

이 설계는 STOMP 시스템을 그대로 유지하면서 raw endpoint만 별도로 끼워 넣기 위한 선택이다. 운영 기본값을 깨지 않고 후보 경로를 테스트할 수 있다.

4.2 인증: access_token query

STOMP는 CONNECT frame header에 Authorization을 실었다. raw WebSocket은 브라우저 WebSocket API에서 임의 header를 넣기 어렵기 때문에 access_token query parameter를 사용한다. JwtHandshakeInterceptor가 handshake 단계에서 token을 파싱해 WebSocket session attributes에 AuthenticatedUser를 저장한다.

상황 정책
토큰 없음 handshake는 허용. principal 없음. PUBLISH는 거부되고 위치/타이핑은 silent ignore.
토큰 파싱 실패 handshake 자체를 401로 거부.
회원 토큰 채팅, 위치, 타이핑 가능.
게스트 토큰 위치, 타이핑 가능. 채팅 PUBLISH는 GUEST_CHAT_NOT_ALLOWED.

4.3 메시지 분기: ChatWebSocketHandler

ChatWebSocketHandler는 raw V2의 중심이다. 들어온 text payload를 InboundFrame으로 파싱하고, Java switch pattern으로 frame type을 분기한다. 

switch (frame) {
  case SubscribeFrame s -> handleSubscribe(session, s);
  case UnsubscribeFrame u -> handleUnsubscribe(session, u);
  case PublishFrame p -> handlePublish(session, p);
  case PositionFrame p -> handlePosition(session, p);
  case TypingFrame t -> handleTyping(session, t);
  case PingFrame ignored -> sendOutbound(session, PongEvent.instance());
}

여기서 중요한 점은 raw V2가 STOMP destination routing을 쓰지 않는다는 것이다. STOMP라면 /app/chat/village destination을 Spring이 보고 handler method를 찾는다. raw V2에서는 우리가 JSON의 type을 보고 직접 method를 호출한다.

4.4 publish 흐름

채팅 PUBLISH는 단순히 WebSocket에 뿌리는 것이 아니다. 기존 채팅과 동일하게 SendMessageUseCase를 호출해 메시지를 저장하고, 그 결과를 MessageEvent로 감싸 Redis bus에 publish한다. 

handlePublish
  -> principal 확인
  -> guest 거부
  -> roomId 가 publicChatRoomId 인지 확인
  -> SendMessageUseCase.execute(...)
  -> MessageEvent.of(...)
  -> bus.publish(roomId, event)

이 때문에 raw V2는 화면상 채팅 기능이 같아 보인다. application use case는 공유하고, transport/broadcast 경로만 달라졌기 때문이다.

4.5 position, typing, disconnect leave

위치와 타이핑은 저장하지 않고 Redis bus에 event를 publish한다. disconnect 시에는 session이 구독했던 room들을 snapshot으로 얻고, 각 room에 POSITION_UPDATE userType: "LEAVE"를 publish한다. 

afterConnectionClosed
  -> roomsOf(sessionId)
  -> bus.publish(roomId, PositionUpdateEvent(... "LEAVE" ...))
  -> unsubscribeAll(sessionId)
  -> sessionRegistry.remove(sessionId)

이 부분은 ghost character 문제와 관련된다. 사용자가 탭을 닫거나 연결이 끊겼을 때 다른 클라이언트가 마지막 좌표에 캐릭터를 계속 두지 않도록 leave event를 보낸다.

5. Redis Pub/Sub이 들어온 위치

Redis는 “채팅 메시지를 저장하는 DB”가 아니다. 여기서 Redis Pub/Sub은 서버 인스턴스 간 실시간 event relay 역할이다. 같은 Spring Boot 서버 안에서만 볼 때는 굳이 Redis가 없어도 fan-out할 수 있다. 하지만 여러 서버 인스턴스를 띄우면 A 서버에 붙은 사용자와 B 서버에 붙은 사용자가 같은 방에 있을 수 있다. 이때 A 서버에서 발생한 메시지를 B 서버에도 알려야 한다.

5.1 단일 서버에서는 왜 체감이 없는가

로컬에서 서버 한 대만 띄우면 Redis relay를 거쳐도 결국 같은 JVM에 다시 돌아온다. 그래서 사용자가 보는 화면은 STOMP와 거의 같다. 체감이 없는 것은 실패가 아니라 당연한 결과다.

5.2 다중 서버에서는 무엇이 달라지는가

Client A -> App Server 1 -> Redis channel chat:room:1
                                  ↓
Client B <- App Server 2 <- Redis channel chat:room:1

핵심은 각 서버가 “내 JVM에 이 방 구독자가 있으면 해당 Redis 채널을 구독한다”는 것이다. 그러면 어떤 서버에서 publish하든 Redis가 해당 채널을 구독 중인 서버들에게 payload를 전달한다. 각 서버는 자기에게 붙어 있는 local WebSocket session들에게만 fan-out한다.

5.3 RoomSubscriptionRegistry의 역할

RoomSubscriptionRegistry는 roomId → local sessionId set을 관리한다. 첫 세션이 방에 들어오면 Redis 채널을 구독하고, 마지막 세션이 나가면 Redis 채널 구독을 해제한다. 

subscribe(roomId, sessionId)
  -> roomToSessions.compute(...)
  -> wasEmpty이면 bus.ensureRoomSubscribed(roomId, this::fanOut)

unsubscribe(roomId, sessionId)
  -> 세션 제거
  -> empty가 되면 bus.removeRoomSubscription(roomId)

이 구조는 “모든 서버가 모든 방 메시지를 받는 구조”를 피한다. 자기 JVM에 구독자가 있는 방만 Redis SUBSCRIBE한다. 이것이 문서에서 말한 O(M×N) 함정 회피의 핵심이다.

5.4 RedisChatRelay의 역할

RedisChatRelay는 RoomMessageBus 구현체다. outbound frame을 JSON으로 직렬화해 Redis channel에 publish하고, 방 구독이 필요하면 RedisMessageListenerContainer에 listener를 등록한다. 

publish(roomId, event)
  -> channel = RoomChannelNaming.chatRoomChannel(roomId)
  -> payload = objectMapper.writeValueAsString(event)
  -> redisTemplate.convertAndSend(channel, payload)

ensureRoomSubscribed(roomId, handler)
  -> roomListeners.computeIfAbsent(...)
  -> listenerContainer.addMessageListener(...)

여기서 설계 원칙은 “방당 1채널 + exact SUBSCRIBE”다. pattern subscribe나 글로벌 채널 하나로 모든 메시지를 받아 앱에서 필터링하는 구조는 부하가 커질 때 위험하다.

6. 프론트 선택 스위치와 확인법

프론트는 화면 컴포넌트가 STOMP/raw를 직접 알지 않도록 realtimeClient.ts facade를 둔다. 화면은 sendVillageMessage, sendPosition, sendTypingStatus 같은 공통 함수를 부른다. facade가 내부에서 STOMP 또는 raw client로 위임한다. 

export function resolveRealtimeTransport(): RealtimeTransport {
  return process.env.NEXT_PUBLIC_REALTIME_TRANSPORT === 'raw' ? 'raw' : 'stomp';
}

7. MSA 대응이 됐다는 말의 정확한 의미

여기서 가장 중요한 오해를 정리해야 한다. 현재 프로젝트가 MSA가 된 것은 아니다. WebSocket 서버가 별도 서비스로 분리된 것도 아니다. REST 서버와 WS 서버가 별도 deploy unit이 된 것도 아니다. 현재는 여전히 하나의 Spring Boot 모놀리식 애플리케이션이다.

“raw V2 + Redis가 들어왔다”는 말은 “MSA 완성”이 아니다. 정확히는 “모놀리식 안에서 서버 간 실시간 relay를 실험할 수 있는 구조가 생겼다”이다.

7.1 이전 구조가 다중 서버에서 막히는 지점

Server 1 SimpleBroker: Server 1에 붙은 sessions만 안다.
Server 2 SimpleBroker: Server 2에 붙은 sessions만 안다.

Server 1에서 /topic/chat/village로 broadcast
  -> Server 1 clients receive
  -> Server 2 clients do not receive unless external broker/relay exists

SimpleBroker는 JVM 내부 broker다. 서버 2대가 있으면 각각 자기 broker와 자기 session registry를 갖는다. 그래서 서버 간 메시지 전달 레이어가 별도로 필요하다.

7.2 raw V2 + Redis가 여는 가능성

Server 1 receives PUBLISH
  -> Redis PUBLISH chat:room:1

Server 1 subscribed to chat:room:1 because it has local clients
  -> fan-out to local clients

Server 2 subscribed to chat:room:1 because it has local clients
  -> fan-out to local clients

이 구조가 되면 WebSocket connection을 어느 서버가 들고 있든 Redis channel을 통해 event를 전달할 수 있다. 이것이 MSA 대응의 기초다. 다만 실제 MSA로 가려면 아직 더 많은 것이 필요하다.

단계 현재 여부 설명
transport 선택 가능 가능 프론트 facade로 STOMP/raw 선택 가능
raw endpoint 존재 가능 /ws/v2 endpoint 등록
Redis fan-out 후보 가능 방 단위 Redis Pub/Sub relay 구현
다중 서버 실측 아직 로컬/CI 테스트는 있지만 실제 2개 앱 인스턴스 검증 필요
WS 전용 서버 분리 아직 별도 module, jar, container, deploy workflow 없음
REST/WS 서비스 계약 분리 아직 인증, 유저 정보, 채팅 저장 use case 호출 경계 재설계 필요

7.3 그래서 지금의 올바른 사고

지금 단계에서 해야 할 사고는 “MSA로 쪼개자”가 아니다. “실시간 계층이 병목이 될 때 쪼갤 수 있는 관측 지점과 교체 가능한 경계가 생겼는가”를 보는 것이다.

  1. STOMP 기본 경로를 유지해 운영 안정성을 보존한다.
  2. raw V2를 선택 가능하게 만들어 같은 기능이 나오는지 확인한다.
  3. Redis fan-out으로 다중 인스턴스 가능성을 검증한다.
  4. 부하 테스트로 병목이 실제로 어디인지 측정한다.
  5. WS 연결 수, 배포 주기, 리소스 격리 요구가 실제로 생기면 분리를 결정한다.

8. 아직 바뀌지 않은 것과 남은 리스크

현재 raw V2 경로는 “STOMP 제거 가능”의 전 단계다. 아직 STOMP를 제거하면 안 되는 이유가 명확히 남아 있다.

항목 현재 상태 왜 중요한가
메일 알림 STOMP /user/queue/mail에 남아 있음 raw V2에는 사용자별 mail notification envelope가 없다.
NPC 응답 broadcast raw V2에 의도적으로 붙이지 않음 현재 BroadcastChatMessagePort는 STOMP adapter가 담당한다.
서버 분리 아직 하지 않음 별도 배포 단위, health check, scaling policy, service contract가 없다.
실제 부하 실측 k6 하네스 준비, 대상 실행 필요 체감이 아니라 latency, error, fan-out 성공률로 판단해야 한다.
reverse proxy 로컬은 보이나 운영 proxy 검증 필요 WebSocket upgrade, timeout, buffering, header 정책이 실제 운영에서 깨질 수 있다.

따라서 현재 결론은 “STOMP 제거”가 아니다. 현재 결론은 “STOMP를 기본값으로 유지하면서 raw V2를 검증 가능한 후보 경로로 둔다”이다.

9. 필요한 CS 지식

이 작업을 제대로 이해하려면 단순히 Spring 코드만 보는 것으로 부족하다. 아래 개념들이 연결되어야 “왜 이 코드가 이런 모양인지”가 보인다.

9.1 WebSocket

  • HTTP handshake 후 TCP connection을 유지한다.
  • 서버가 클라이언트에게 push할 수 있다.
  • 브라우저 WebSocket API는 임의 Authorization header를 넣기 어렵다.
  • 그래서 raw V2는 query token 또는 cookie 같은 다른 인증 전달 방식을 고민해야 한다.

9.2 STOMP

  • WebSocket 위에서 동작하는 messaging protocol이다.
  • CONNECT, SUBSCRIBE, SEND, MESSAGE, ERROR 같은 frame이 있다.
  • destination 기반 routing을 제공한다.
  • Spring과 잘 붙지만, 내부 SimpleBroker를 쓰면 scale-out 한계가 있다.

9.3 Pub/Sub

  • publisher는 channel에 메시지를 보낸다.
  • subscriber는 channel을 구독하고 메시지를 받는다.
  • Redis Pub/Sub은 메시지를 저장하지 않는다. subscriber가 그 순간 없으면 메시지는 사라진다.
  • 따라서 채팅 영속 저장은 Cassandra/PostgreSQL 쪽 use case가 담당하고, Redis는 실시간 relay만 담당한다.

9.4 Fan-out

  • 하나의 메시지를 여러 수신자에게 복제해서 보내는 작업이다.
  • 채팅방 인원이 늘면 fan-out 비용이 커진다.
  • 서버 N대, 방 M개, 유저 K명이 있을 때 어떤 서버가 어떤 방을 구독할지 설계가 중요하다.
  • 모든 서버가 모든 메시지를 받으면 불필요한 CPU/네트워크 비용이 커진다.

9.5 Backpressure와 느린 클라이언트

  • WebSocket은 연결이 유지되므로 느린 클라이언트에게 메시지가 쌓일 수 있다.
  • 위치 frame은 고빈도라 오류 응답을 매번 보내면 클라이언트 큐가 폭발할 수 있다.
  • 그래서 raw V2는 principal 없음, NaN/Infinity 좌표 같은 고빈도 오류를 silent ignore한다.

9.6 상태와 세션

  • WebSocket session은 특정 서버 인스턴스 메모리에 있다.
  • 서버가 여러 대면 session registry는 전역이 아니라 local이다.
  • Redis는 event를 전달할 수 있지만 실제 socket write는 해당 session을 가진 서버가 해야 한다.

9.7 모놀리식과 MSA

  • 모놀리식은 하나의 배포 단위 안에 여러 기능이 있는 구조다.
  • MSA는 배포 단위와 데이터/운영 책임을 분리한다.
  • 코드 패키지를 나눈다고 MSA가 되는 것이 아니다.
  • Redis를 넣었다고 MSA가 되는 것도 아니다.
  • 서비스 간 계약, 배포, 장애 격리, 관측, 데이터 소유권이 분리되어야 한다.

10. 탑다운 학습 로드맵

아래 순서로 보면 현재 프로젝트를 “화면 → 프론트 선택 → 백엔드 endpoint → application use case → broadcast → scale-out” 순서로 이해할 수 있다.

Level 1. 화면에서 시작

  1. 브라우저에서 채팅을 보낸다.
  2. Network → WS에서 연결 URL을 확인한다.
  3. Frames에서 STOMP frame인지 raw JSON인지 본다.

목표: “같은 화면인데 내부 frame이 다르다”를 체감한다.

Level 2. 프론트 facade 읽기

  1. useStomp.ts에서 connect lifecycle을 본다.
  2. realtimeClient.ts에서 transport 선택을 본다.
  3. stompClient.ts와 rawWebSocketClient.ts의 send 함수를 나란히 비교한다.

목표: UI 컴포넌트가 STOMP/raw를 직접 알지 않도록 분리된 구조를 이해한다.

Level 3. STOMP 백엔드 읽기

  1. WebSocketConfig.java에서 endpoint, broker, prefix를 본다.
  2. ChatMessageHandler.java에서 @MessageMapping을 본다.
  3. PositionHandler.java, TypingHandler.java에서 broadcast destination을 본다.

목표: Spring STOMP가 routing과 broker 역할을 얼마나 대신하는지 이해한다.

Level 4. raw V2 백엔드 읽기

  1. WebSocketV2Config.java에서 handler mapping을 본다.
  2. JwtHandshakeInterceptor.java에서 query token 인증을 본다.
  3. ChatWebSocketHandler.java에서 frame switch를 본다.
  4. InboundFrame.java와 frame records에서 JSON protocol을 본다.

목표: STOMP가 해주던 protocol routing을 우리 코드가 직접 처리한다는 것을 이해한다.

Level 5. Redis relay 읽기

  1. RoomSubscriptionRegistry.java에서 room → local session set을 본다.
  2. RedisChatRelay.java에서 publish/subscribe를 본다.
  3. RedisChatRelayTest와 ChatWebSocketV2IntegrationTest에서 검증 의도를 본다.

목표: 여러 서버가 있을 때 메시지가 어떻게 이동할 수 있는지 이해한다.

Level 6. MSA 판단 기준 세우기

  1. 현재 병목이 WebSocket connection 수인지, DB인지, NPC 응답인지, frontend 렌더링인지 구분한다.
  2. 부하 테스트에서 p95/p99 latency, error count, Redis CPU, JVM thread, GC, socket count를 본다.
  3. 운영에서 WS와 HTTP 배포 주기가 충돌하는지 본다.
  4. 그때 WS 서버 분리, broker 교체, sticky session, autoscaling 정책을 결정한다.

목표: “MSA가 멋져서 분리”가 아니라 “측정된 병목과 운영 요구 때문에 분리”라는 사고를 갖는다.

11. 직접 확인 체크리스트

11.1 raw 모드 확인

  1. PUBLIC_REALTIME_TRANSPORT=raw로 프론트를 다시 빌드한다.
  2. 브라우저 Network → WS에서 /ws/v2 연결을 확인한다.
  3. Frames에서 SUBSCRIBE, POSITION, TYPING, PUBLISH JSON을 확인한다.
  4. 채팅 메시지를 보내고 MESSAGE 수신을 확인한다.
  5. 다른 브라우저 또는 시크릿 창을 열어 위치와 채팅이 서로 보이는지 확인한다.

11.2 STOMP 모드와 비교

  1. PUBLIC_REALTIME_TRANSPORT를 비우거나 stomp로 둔다.
  2. 프론트를 다시 빌드한다.
  3. Network → WS에서 /ws/.../websocket 경로를 확인한다.
  4. Frames에서 STOMP SEND, SUBSCRIBE frame을 확인한다.

11.3 k6 raw smoke

BASE_URL=http://localhost:8080 COUNT=10 node loadtest/prepare-tokens.js

BASE_URL=http://localhost:8080 RAW_WS_URL=ws://localhost:8080/ws/v2 \
  k6 run --vus 1 --duration 30s loadtest/raw-v2-mixed.js

이 테스트는 화면 검증이 아니다. handshake, position, typing, chat 송수신이 실제 raw protocol로 되는지 보는 하네스다.

11.4 “개선됐는지”를 판단하는 지표

지표 왜 보는가
WebSocket handshake 성공률 proxy, CORS, 인증 query가 정상인지 확인

position sent/received 비율 고빈도 fan-out 누락 확인

마지막 정리

이 작업은 사용자에게 즉시 보이는 기능 개선 작업이 아니다. 대신 실시간 통신 경로를 운영 기본 STOMP에서 raw WebSocket + Redis 후보 경로로 교체 가능한 상태로 만들고, 그 교체가 실제로 안전한지 검증할 수 있는 발판을 만든 작업이다.

지금 이 프로젝트는 여전히 모놀리식이다. 그러나 이전보다 더 좋은 상태다. 이유는 “쪼갰기 때문”이 아니라 “쪼갤지 말지를 판단할 수 있는 경계와 검증 도구가 생겼기 때문”이다.

다음에 해야 할 일은 감으로 MSA를 선택하는 것이 아니다. raw V2 경로로 smoke/load를 실행하고, 브라우저 수동 검증과 부하 지표를 보고, 메일 알림과 NPC 응답까지 raw 쪽으로 옮길지 결정하는 것이다.
