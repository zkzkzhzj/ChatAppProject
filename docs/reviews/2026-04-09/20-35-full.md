## Codex 리뷰 결과

프로젝트에는 동시성/멱등성 관련 Critical Rules 위반과 Kafka 소비 실패 시 이벤트 유실로 이어지는 문제가 있습니다. 특히 회원가입과 village 초기화 경로는 실제 운영 환경에서 잘못된 500 응답이나 중복 데이터 생성을 유발할 수 있습니다.

---

### [P1][CRITICAL] 이메일 중복 검사 레이스 컨디션 제거

**파일:** `backend/src/main/java/com/maeum/gohyang/identity/application/service/RegisterUserService.java:29-30`

동일 이메일로 동시 회원가입 요청이 들어오면 두 요청이 모두 `isEmailTaken()` 검사를 통과하고 `saveWithLocalAuth()`로 진행할 수 있다. 두 번째 insert는 DB 유니크 제약에 걸려 의도한 `DuplicateEmailException` 대신 500 에러가 클라이언트에 반환된다. AGENTS.md가 명시적으로 금지한 check-then-act 패턴이다.

---

### [P1][CRITICAL] Kafka 소비 실패 시 재시도 가능하도록 예외 재던지기

**파일:** `backend/src/main/java/com/maeum/gohyang/village/adapter/in/messaging/UserRegisteredEventConsumer.java:61-66`

JSON 파싱, village 초기화, `processed_event` 저장 중 실패하면 현재 catch 블록이 로그와 알림만 남기고 정상 리턴한다. Kafka 리스너에서 정상 리턴은 해당 레코드가 처리 완료된 것으로 간주되므로, `user.registered` 이벤트가 영구적으로 유실되어 해당 유저의 village가 생성되지 않는다. 일시적 DB 장애 시 재시도했다면 복구 가능한 케이스다.

---

### [P1][CRITICAL] Village 초기화 guard를 원자적 방식으로 교체

**파일:** `backend/src/main/java/com/maeum/gohyang/village/application/service/InitializeUserVillageService.java:28-34`

현재 중복 방지 guard가 이후 두 개의 insert와 비원자적으로 실행된다. at-least-once 전달 또는 동시 호출 시 두 실행 모두 character가 없다고 판단하고 기본 village를 동시에 생성할 수 있다. `space` 테이블에 `(userId, isDefault)` 유니크 제약이 없으면 중복 기본 공간이 저장된다. AGENTS.md가 isPresent()-then-act 패턴을 금지하는 이유가 정확히 이것이다.

---

### [P2][WARNING] STOMP 메시지 페이로드 검증 누락

**파일:** `backend/src/main/java/com/maeum/gohyang/communication/adapter/in/websocket/ChatMessageHandler.java:34-40`

REST 엔드포인트는 채팅 메시지에 `@NotBlank` / `@Size(max = 1000)` 검증을 적용하지만, WebSocket 경로는 raw `StompSendMessageRequest`를 bean validation 없이 수락한다. 클라이언트가 STOMP로 빈 메시지나 초과 크기 메시지를 보내면 그대로 저장·브로드캐스트되어 REST API 계약을 우회한다.
