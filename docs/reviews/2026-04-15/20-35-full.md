## Codex 리뷰 결과 (원문)

> The project has confirmed event-loss and cross-store consistency problems in its Kafka and chat flows, plus a registration race that converts duplicate signups into 500s. These are operationally significant enough that the code should not be considered correct as-is.

### Codex 발견 항목 (P1 x 3)

1. **[P1] Kafka exhausted records 유실** — `KafkaConsumerConfig.java:24-26`
   DefaultErrorHandler에 recoverer 없이 FixedBackOff(3회)만 설정. 3회 재시도 소진 시 레코드가 영구 스킵됨. 회원가입 후 village 초기화 누락, 대화 요약 중단 가능.

2. **[P1] 회원가입 check-then-insert 레이스** — `RegisterUserService.java:31-40`
   `isEmailTaken()` 확인 후 insert 사이에 동시 요청이 끼어들면 unique constraint violation이 500으로 노출됨.

3. **[P1] Cassandra-PostgreSQL 교차 저장소 비정합** — `SendMessageService.java:61-66`
   Cassandra에 메시지를 먼저 쓴 후 PostgreSQL 트랜잭션이 롤백되면 Cassandra 메시지는 남아있어 중복 메시지 노출 가능.

### [CRITICAL] 아키텍처 위반

#### C-01. check-then-act 멱등성 패턴 위반 (Critical Rule #6)
- **RegisterUserService.java:31-33** — `isEmailTaken()` 확인 후 `saveWithLocalAuth()` 호출. 동시 요청 시 두 요청 모두 통과 가능. `DataIntegrityViolationException`을 `DuplicateEmailException`으로 변환하지 않아 500 응답 노출.
- **InitializeUserVillageService.java:30** — `loadCharacterPort.load(userId).isPresent()` 확인 후 insert. processed_event 테이블이 1차 방어선이라고 주석에 명시했지만, isPresent check-then-act 자체가 AGENTS.md Critical Rule #6 위반. `INSERT ... ON CONFLICT DO NOTHING` 기반으로 전환 필요.

#### C-02. IdempotencyGuard check-then-act 패턴 (Critical Rule #6)
- **IdempotencyGuard.java:20** + **UserRegisteredEventConsumer.java:48** + **ConversationSummaryEventConsumer.java:58** — `existsByEventId()` 확인 후 처리, 완료 후 `markAsProcessed()` 호출. 이 사이에 동일 이벤트가 다시 도착하면 중복 처리 발생. `insertIfAbsent` (INSERT ON CONFLICT DO NOTHING) 원자적 패턴으로 전환 필요.

#### C-03. Kafka exhausted record 유실 (운영 리스크)
- **KafkaConsumerConfig.java:24-26** — `DefaultErrorHandler`에 `DeadLetterPublishingRecoverer` 등 recoverer가 없음. 주석에 "DLT는 모니터링 인프라 구축 후 추가"라고 명시되어 있지만, 현재 상태에서 3회 재시도 소진 시 이벤트 영구 유실. `user.registered` 이벤트 유실 시 캐릭터/공간 미생성, `npc.conversation.summarize` 유실 시 대화 요약 영구 중단.

#### C-04. 교차 저장소 비정합 (데이터 정합성)
- **SendMessageService.java:61-64** — Cassandra에 `saveWithUser()` 호출 후 line 66에서 PostgreSQL outbox 발행. PostgreSQL 트랜잭션 롤백 시 Cassandra 커밋은 되돌릴 수 없어 유저에게 중복 메시지가 보임. Cassandra 쓰기를 트랜잭션 성공 후(커밋 후)로 이동하거나, 실패 시 보상 트랜잭션 필요.

#### C-05. global 패키지에서 identity adapter 직접 참조 (도메인 간 참조 위반 방향)
- **StompAuthChannelInterceptor.java:14** — `import com.maeum.gohyang.identity.adapter.in.security.JwtProvider`. AGENTS.md security 패키지 분리 규칙에 따르면 global/config에서 identity/adapter를 직접 import하는 것은 의존 방향 위반. JwtProvider가 구현하는 Port 인터페이스를 global에 두고 그것을 참조하거나, STOMP 인터셉터를 identity/adapter/in/security/로 이동 필요.

### [WARNING] 컨벤션 위반

### LGTM

- **Domain Entity 순수 POJO** — ChatRoom, Message, Participant, User, Character, Space 모두 인프라 어노테이션 없이 순수 POJO로 구현. 정적 팩토리 메서드(`newXxx()`, `restore()`) 패턴 일관 적용.
- **@Autowired 필드 주입 없음** — 전체 코드베이스에서 `@Autowired` 0건. 모두 `@RequiredArgsConstructor` 생성자 주입.
- **throw new RuntimeException() 없음** — 전체 코드베이스에서 0건. 모든 예외가 커스텀 예외(domain/error/) 사용.
- **도메인 간 직접 참조 없음** — identity, communication, village 간 Entity/Repository 직접 import 없음. userId 같은 ID 값으로만 연결.
- **Outbox 패턴 완성도** — 에러 분류(transient/permanent/systemic), retry 카운터, AlertPort 알람까지 잘 구현됨.
- **SendMessageService.getOrCreateParticipant()** — `DataIntegrityViolationException` catch 후 재조회하는 upsert 패턴으로 동시성 처리. 잘 구현됨.
- **Kafka Consumer 멱등성 구조** — IdempotencyGuard + Outbox eventId 헤더 조합으로 중복 처리 방어 (check-then-act 원자성 이슈는 별도 지적).
- **DTO Validation** — REST Request DTO에 `@NotBlank`, `@Size`, `@Email` 적용 일관.
- **compareAndSet 루프** — SendMessageService의 메시지 카운터가 원자적 CAS 루프로 구현되어 동시성 안전.
