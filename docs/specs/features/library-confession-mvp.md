# 도서관 고백 기록 MVP

> 이 spec 은 트랙 `library-confession-mvp` (Issue #116) 의 요구사항 진실이다.
> 진행 상태는 `docs/handover/track-library-confession-mvp.md`, 제품 컨셉은
> [도서관 고백 기록 공간](../../knowledge/library-confession-room.md), 구현 계획은
> [도서관 고백 기록 MVP 구현 계획](../../planning/library-confession-mvp-plan.md)을 따른다.

## 1. Outcomes

- 사용자는 마을 도서관에서 익명 고백 기록을 남길 수 있다.
- 사용자는 책장에서 고백 기록을 꺼내 읽는 느낌으로 공개 고백 기록을 열람할 수 있다.
- 사용자는 고백 작성자에게만 보이는 비공개 편지를 남길 수 있다.
- 고백 작성자는 내게 온 편지를 확인하고 편지별 감사 답장 1회를 보낼 수 있다.
- 공개 API와 화면에는 고백 작성자, 편지 작성자 식별 정보가 노출되지 않는다.
- NPC는 공개 고백 기록만 참조해 태그 기반 유사 기록을 안내한다.
- 위험 고백 감지, 신고, 숨김을 위한 최소 정책 진입점이 생긴다.

## 2. Scope

### 2.1 In

- 고백 기록 작성, 목록, 상세, 삭제
- 책장/태그 기반 고백 기록 분류
- 비공개 편지 작성, 수신 편지 조회, 내가 남긴 편지 조회
- 편지별 감사 답장 1회
- 이모지 공감과 간략 집계
- 태그 기반 NPC 유사 고백 기록 안내
- 위험 고백 감지 상태와 신고/숨김을 위한 최소 모델
- 성공/실패 케이스 테스트

### 2.2 Out

- 공개 댓글, 대댓글, 토론 스레드
- 실시간 상담 채팅
- 인기순 랭킹, 홍보 피드, 공유 성장 루프
- embedding/vector search
- 로컬 LLM 기반 NPC 응답 생성
- 운영자 고급 콘솔
- 비공개 편지의 NPC 학습/요약/인용

## 3. Constraints

| 차원 | 제약 |
|------|------|
| 아키텍처 | Hexagonal Architecture를 따른다. Domain은 Application/Adapter를 모른다. |
| 익명성 | 외부 응답에 `userId`, email, provider id, 프로필 식별자를 노출하지 않는다. |
| 비공개성 | 편지는 수신자인 고백 작성자와 편지 작성자 본인 외에는 조회할 수 없다. |
| 멱등성 | 감사 답장 1회, 이모지 중복 방지는 `exists()` 후 처리하지 않는다. |
| 안전 | `HIGH` 이상 위험 고백은 NPC 일반 안내 대상에서 제외하거나 제한한다. |
| MVP | 검색/LLM/운영자 콘솔은 확장 지점만 남기고 구현하지 않는다. |

## 4. Decisions

### D1. [제품 구조] 고백 기록과 토론 기록은 분리한다

- **왜**: 고백 기록은 익명, 편지, 조용한 열람이 핵심이고 토론 기록은 공개 의견 교환이 핵심이라 권한과 UX가 다르다.
- **대안**:
  - 하나의 게시글 타입에 옵션만 추가 — 권한 규칙이 섞이고 나중에 공개 댓글이 고백 경험을 침범할 수 있다.
- **빈틈**: 토론 기능을 빨리 붙이고 싶을 때 중복 모델처럼 보일 수 있다.
- **재검토 트리거**: 토론형 게시글 구현 트랙이 시작될 때 공통 추상화 여부를 다시 본다.

### D2. [인증·인가] 완전 익명 표시, 내부 userId 보관

- **왜**: 사용자에게는 완전 익명 경험을 제공하되 삭제, 신고, 편지 조회 권한을 위해 서버 내부 식별자는 필요하다.
- **대안**:
  - 서버도 작성자를 저장하지 않음 — 삭제/편지/신고/악용 대응이 불가능하다.
  - 익명 별칭 표시 — 재식별 단서가 될 수 있다.
- **빈틈**: 운영자 권한 설계가 약하면 내부 식별자가 노출될 수 있다.
- **재검토 트리거**: 운영자 콘솔 또는 감사 로그 구현 시.

### D3. [API 설계] 공개 고백 API와 내 편지 API를 분리한다

- **왜**: 공개 고백 상세에 비공개 편지가 섞이면 권한 실수와 데이터 노출 위험이 커진다.
- **대안**:
  - 고백 상세 응답에 작성자용 편지도 조건부 포함 — DTO와 권한 분기가 복잡해진다.
- **빈틈**: 클라이언트 호출 수가 늘어난다.
- **재검토 트리거**: 편지함 UX 성능 문제가 확인될 때.

### D4. [멱등성] 감사 답장과 공감 중복은 DB 제약으로 막는다

- **왜**: `exists()` 확인 후 insert는 check-then-act 경쟁 조건을 만든다.
- **대안**:
  - 애플리케이션 레벨 중복 검사 — 동시 요청에서 깨질 수 있다.
- **빈틈**: DB별 upsert/unique 충돌 처리 매핑이 필요하다.
- **재검토 트리거**: DB 마이그레이션 또는 테스트에서 충돌 처리 일관성이 깨질 때.

### D5. [NPC 정책] NPC는 공개 고백 기록만 참조한다

- **왜**: 편지는 작성자에게만 닿는 사적 말이므로 NPC 요약/학습/인용 대상이 되면 신뢰를 잃는다.
- **대안**:
  - 편지도 익명화 후 활용 — 동의/철회/감사 로그 정책이 필요해 MVP 범위를 넘는다.
- **빈틈**: NPC 안내 품질이 초기에는 단순할 수 있다.
- **재검토 트리거**: 사용자가 편지 활용에 명시 동의하는 정책과 데이터 삭제 정책이 생길 때.

## 6. Verification

- [ ] 고백 기록 작성/목록/상세/삭제 성공 케이스가 통과한다.
- [ ] 공개 응답에 작성자 식별 정보가 포함되지 않는다.
- [ ] 비작성자는 고백 작성자에게 온 편지를 조회할 수 없다.
- [ ] 편지 작성자는 자신이 남긴 편지를 조회할 수 있다.
- [ ] 감사 답장은 편지별 1회만 생성된다.
- [ ] 이모지 공감 중복 처리는 DB 제약 또는 insert-if-absent로 보장된다.
- [ ] NPC 유사 기록 조회에 비공개 편지와 감사 답장이 포함되지 않는다.
- [ ] 위험 고백 감지 결과가 NPC 일반 안내와 신고/검토 정책에 연결된다.
- [ ] 새 기능 성공/실패 테스트가 존재한다.
