# 사서방 프로토타입 설계

## 목표

도서관 내부 경험을 전역 UI 패널이 아니라 사서방 안의 공간 상호작용으로 재구성한다. 사용자는 사서 NPC, 책장, 우편 알림을 통해 고민 상담, 도서 남기기, 타인의 기록 열람, 마음 보내기, 답장 알림을 분리해서 경험한다.

## 비목표

- 실제 LLM/RAG 파이프라인 완성은 이번 범위가 아니다.
- 우편 알림에서 편지나 답장 전문을 모두 보여주지 않는다.
- 책장 열람의 완성형 3D 책 애니메이션은 이번 범위가 아니다. 다만 게임식 책장 확대와 책 펼침 감각은 프로토타입에 포함한다.
- 기존 confession/letter/reply 백엔드 도메인을 새 도메인으로 갈아엎지 않는다.

## 사서방 공간 구조

마을에서 기존 도서관 위치로 진입하면 내부 명칭은 일단 사서방으로 취급한다. 좌측 하단의 책 버튼은 제거한다.

사서방 안에는 두 개의 주요 상호작용 지점이 있다.

- 사서 NPC: 앉아 있는 NPC. 가까이 가면 링 버튼이 나타난다.
- 책장: 뒤쪽 벽면을 채운 책장. 가까이 가면 열람 상호작용이 나타난다.

상호작용 UI는 사용자가 해당 오브젝트 근처에 있을 때만 보인다. 사서방 밖에서는 관련 UI가 보이지 않는다.

## 사서 NPC 흐름

NPC 근처에서 링 버튼을 누르면 사서 대화 패널이 열린다.

첫 선택지는 다음 두 가지다.

- 고민 상담하기
- 도서 남기기

고민 상담하기는 프로토타입에서 실제 LLM/RAG 대신 기존 유사 기록 조회 또는 임시 응답을 사용한다. 사서는 "비슷한 마음이 남겨져 있었어요" 같은 톤으로 기존 기록을 바탕으로 가볍게 답한다.

도서 남기기는 기존 고백 작성 API를 재사용한다. 화면 문구는 게시가 아니라 사서에게 도서를 맡기는 감각으로 표현한다. 제출 후 사서가 받았다는 짧은 응답을 보여주고, 해당 기록은 책장 열람 목록에 들어간다.

## 책장 열람 흐름

책장 근처에서 상호작용하면 평면 리스트가 바로 뜨지 않는다. 프로토타입은 다음 순서를 따른다.

1. 화면이 책장 쪽으로 확대되는 듯한 전환을 보여준다.
2. 책 제목들이 꽂힌 책장형 패널을 보여준다.
3. 좌우 버튼으로 페이지를 넘기며 개수 제한 없이 탐색할 수 있게 한다.
4. 책 제목을 클릭하면 책을 꺼내는 짧은 모션 뒤 펼쳐진 책 상세 화면으로 전환한다.
5. 상세 마지막에는 닫기와 마음 보내기를 제공한다.

첫 구현은 CSS 전환과 패널 상태 전환으로 만든다. Three.js 카메라 이동이나 개별 3D 책 오브젝트는 후속 고도화로 둔다.

## 마음 보내기

기존 편지쓰기 기능은 책 상세 화면의 마음 보내기로 이름과 흐름을 바꾼다. 사용자가 타인의 도서를 읽은 뒤에만 자연스럽게 마음을 보낼 수 있게 한다.

마음 보내기 성공 후에는 전문을 다시 노출하지 않고, 보냈다는 짧은 피드백만 보여준다.

## 우편 알림

우측 하단 말풍선 UI 위쪽에 작은 우편 아이콘을 둔다. 다음 이벤트가 있으면 간단 알림을 보여준다.

- 내가 남긴 도서에 마음이 도착함
- 내가 보낸 마음에 답장이 도착함

우편 알림은 전체 본문을 보여주지 않는다. 보낸 사람 식별 정보도 노출하지 않는다. 알림은 짧은 상태 확인 용도이며, 후속에서 책장 열람 안의 탭으로 내가 남긴 도서와 내가 보낸 마음을 모아볼 수 있게 구조를 열어둔다.

## 프론트엔드 구조

기존 `LibraryOverlay`는 좌측 하단 전역 패널 역할을 제거하고 사서방 상호작용 컨테이너로 재구성한다.

필요한 UI 단위는 다음과 같다.

- `LibrarianInteraction`: NPC 링 버튼과 대화/도서 남기기 패널
- `BookshelfInteraction`: 책장 접근 버튼, 책장 확대 패널, 책 상세 펼침 화면
- `MailNotification`: 우측 하단 우편 알림

파일 분리는 구현 중 기존 코드 크기를 보고 결정하되, 한 파일이 커지면 위 단위로 분리한다.

## Three.js 연동

`LibraryScene`은 NPC와 책장의 상호작용 기준점을 가진다. `SceneManager`는 매 프레임 현재 캐릭터가 가까운 대상이 있는지 계산해서 React 쪽으로 전달한다.

전달 상태는 최소한 다음 값을 포함한다.

- 현재 scene: `village`, `library`, `transitioning`
- 가까운 사서 NPC 여부
- 가까운 책장 여부

사서방 밖에서는 상호작용 상태를 비운다.

## 데이터 흐름

기존 confession API를 우선 재사용한다.

- 도서 목록: confession 목록 조회
- 도서 상세: confession 상세 조회
- 도서 남기기: confession 생성
- 마음 보내기: confession letter 생성
- 답장 확인: thank reply 조회
- 사서 상담: NPC 유사 기록 조회 또는 임시 응답

프로토타입에서는 서버 API 계약을 크게 변경하지 않는다. 문구와 UI 메타포를 프론트에서 사서방 경험에 맞춘다.

## 오류 처리

회원 전용 작업에서 토큰이 없거나 회원이 아니면 기존 로그인 유도 흐름을 사용한다.

API 실패 시 사서/NPC 톤의 짧은 오류 메시지를 보여준다. 예를 들면 "지금은 책장을 열기 어려워요. 잠시 뒤 다시 와 주세요."처럼 사용자가 다음 행동을 알 수 있게 한다.

## 테스트

프론트 테스트는 최소 다음을 확인한다.

- 사서방 scene이 아닐 때 사서방 UI가 노출되지 않는다.
- 사서 NPC 근처 상태에서 NPC 상호작용 버튼이 보인다.
- 책장 근처 상태에서 책장 상호작용 버튼이 보인다.
- 책장 흐름에서 책장 패널, 책 상세, 닫기, 마음 보내기 상태 전환이 동작한다.
- 우편 알림은 전문을 노출하지 않는다.

백엔드 변경이 없다면 백엔드 테스트는 추가하지 않는다. API 계약을 바꾸는 경우에는 해당 도메인 서비스/컨트롤러 테스트를 추가한다.

## 성공 기준

- 좌측 하단 책 버튼이 사라진다.
- 사서방 안에서만 NPC와 책장 상호작용이 가능하다.
- NPC를 통해 상담과 도서 남기기 흐름에 진입할 수 있다.
- 책장은 리스트가 아니라 책장 확대, 좌우 탐색, 책 꺼내기, 펼쳐 읽기 감각을 가진다.
- 책 상세에서 마음 보내기가 가능하다.
- 우측 하단 우편 아이콘에서 간단 알림만 확인할 수 있다.
