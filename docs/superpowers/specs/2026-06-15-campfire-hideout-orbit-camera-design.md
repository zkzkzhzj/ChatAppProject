# 모닥불 아지트 + 오빗 카메라 디자인

> 날짜: 2026-06-15
> 범위: `frontend/src/three`
> 상태: 사용자 리뷰 준비

## 1. 목표

마을을 사용자가 들어오고 싶고, 머무르고 싶고, 기억할 수 있는 장소로 만든다.

이번 디자인의 정체성은 단순히 "귀여운 숲"이 아니다. `마음의 고향`은 사람들이 중앙
모닥불 주변에 자연스럽게 모여 이야기하는 따뜻한 숲속 아지트여야 한다. 첫 PR에서는
중앙 모닥불을 마을의 얼굴로 강화하고, 마우스로 캐릭터 주변을 360도 둘러볼 수 있는
카메라를 추가한다.

## 2. 비목표

- 마을 전체 맵을 갈아엎지 않는다.
- 이번 PR에서 새 공개 GLB 에셋을 추가하지 않는다.
- 캐릭터 모델을 다시 디자인하지 않는다.
- 채팅, 편지함, 읽음 처리 동작을 바꾸지 않는다.
- 편지함 중복 문구 제거는 이번 PR에 포함하지 않는다.
- 이번 PR에서 이동 방향을 카메라 기준 이동으로 바꾸지 않는다.
- 회귀 방지에 필요한 경우가 아니라면 모바일 카메라 제스처 전체 설계를 추가하지 않는다.

## 3. 디자인 방향

### 분위기

- 숲속 아지트
- 따뜻하고, 조용하고, 귀엽고, 들어가고 싶은 느낌
- 장식용 공원보다 앉아서 이야기하고 싶은 장소

### 시그니처 영역

중앙 모닥불을 마을의 랜드마크로 만든다.

현재의 불, 돌, 그루터기 의자 구조는 기반으로 유지할 수 있다. 다만 주변 배치가 의도된
원형 대화 공간처럼 읽혀야 한다. 사용자는 입장 후 몇 초 안에 "여기가 사람들이 모여
이야기하는 자리"라는 것을 알아야 한다.

### 정체성 단서

정체성은 설명 UI 텍스트가 아니라 오브젝트와 구도로 전달한다.

허용하는 단서:

- 모닥불 주변의 원형 흙길 또는 닳은 잔디 링
- 불을 바라보는 통나무 의자 또는 그루터기 의자
- 의자 근처의 낮은 테이블, 나무 상자, 그루터기 받침
- 대화 공간 주변의 낮은 랜턴 기둥 또는 매달린 랜턴
- 모닥불 근처의 따뜻한 빛 입자
- 원형 공간 근처의 작은 나무 표지판 또는 기념 소품
- 공간이 편지 기능처럼 보이지 않는 선에서만 아주 은근한 편지/고향 상징

피해야 할 것:

- 큰 텍스트 라벨
- 축제장처럼 보일 정도의 과한 장식
- 영역을 산만하게 만드는 에셋 중심 장식
- 분위기를 설명하는 UI 카드나 튜토리얼 문구

## 4. 카메라 디자인

현재 고정 follow 카메라를 orbit-follow 카메라로 바꾼다.

### 동작

- 카메라 타깃은 로컬 캐릭터를 유지한다.
- 마우스 드래그로 캐릭터 주변 yaw를 360도 회전할 수 있다.
- pitch는 제한해서 땅 밑을 보거나 장면을 잃어버리는 시야를 막는다.
- 첫 PR에서는 카메라 거리를 고정하거나 아주 좁은 범위에서만 제한한다.
- 카메라 보간은 현재의 차분한 움직임과 맞게 부드럽게 유지한다.
- 기존 키보드 이동 의미는 이번 PR에서 바꾸지 않는다.

### 입력 범위

데스크톱:

- 게임 캔버스 위 primary pointer drag로 카메라를 회전한다.
- 우클릭이나 컨텍스트 메뉴 동작이 오빗에 필수적이면 안 된다.
- 텍스트 입력 위에서 드래그할 때는 카메라가 회전하면 안 된다.

모바일:

- 기존 가상 조이스틱을 우선한다.
- 안전하게 충돌 없이 넣을 수 있는 경우가 아니라면 첫 PR에서 전체 모바일 오빗 제스처는 제외한다.

## 5. 아키텍처 메모

변경은 기존 Three.js 경계 안에 둔다.

예상 파일:

- `frontend/src/three/constants.ts`: 카메라 오빗 상수
- `frontend/src/three/input.ts`: 필요한 경우 포인터/카메라 입력 상태
- `frontend/src/three/SceneManager.ts`: 카메라 입력 연결과 업데이트 흐름
- `frontend/src/three/scenes/VillageScene.ts`: orbit-follow 카메라 동작
- `frontend/src/three/scenes/villageDecor.ts`: 모닥불 아지트 장식
- 기존 input/decor 테스트 근처의 집중 테스트

기존 `SceneManager`와 `VillageScene` 경계로 충분히 처리할 수 있다면 새 카메라 프레임워크나
큰 추상화를 만들지 않는다.

## 6. 완료 조건

- 중앙 모닥불 영역이 주변 장식보다 강한 시각적 중심으로 보인다.
- 모닥불이 원형 좌석과 따뜻한 조명으로 대화 아지트처럼 읽힌다.
- 추가 소품 배치는 결정적이며 `Math.random()`을 사용하지 않는다.
- 사용자는 마우스 입력으로 캐릭터 주변 카메라를 360도 회전할 수 있다.
- pitch와 distance가 제한되어 깨진 시야로 들어가지 않는다.
- 기존 WASD와 가상 조이스틱 이동이 계속 동작한다.
- 채팅 입력 포커스가 게임 입력 충돌을 막는다.
- 새 공개 에셋 라이선스 표면을 만들지 않는다.
- 새 카메라/입력 동작 또는 비자명한 데코 배치 로직은 테스트로 확인한다.

## 7. 검증 계획

- Three.js 입력, 마을 데코, 장면 동작과 관련된 집중 프론트엔드 테스트를 실행한다.
- 프론트엔드 TypeScript 검증을 실행한다.
- 로컬 앱을 실행하거나 직접 확인해서 다음을 본다.
  - 모닥불 영역이 마을의 정체성으로 바로 읽히는지
  - 마우스 오빗이 동작하는지
  - 이동이 계속 동작하는지
  - 채팅 입력이 방해받지 않는지
  - 모바일 조이스틱이 계속 사용할 수 있는지

## 8. 리스크

- 360도 카메라 오빗은 현재 맵의 약한 면을 노출할 수 있다. 첫 PR은 모닥불 영역을 강화하되,
  마을 전체의 모든 각도가 같은 수준으로 완성되었다고 약속하지 않는다.
- 오빗이 들어가면 카메라 기준 이동이 기대될 수 있다. 하지만 카메라, 이동, 모바일, 위치 동기화를
  한 변경에 묶지 않기 위해 의도적으로 다음 단계로 미룬다.
- 장식을 늘리면 고유 geometry/material이 많아져 성능에 부담이 될 수 있다. 가능한 한 단순한
  primitive mesh와 재사용 material을 쓴다.
